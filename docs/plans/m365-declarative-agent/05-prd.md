## 一、产品概述

**一句话定义**：SocialHub Declarative Agent 是一个部署在 Microsoft 365 Copilot 中的客户数据智能助手，让企业运营和管理团队在 Teams 日常工作流内直接用自然语言查询客户健康指标，无需切换 CLI 或 BI 系统。

### 目标

| 维度 | 目标 |
|---|---|
| 用户价值 | 把 SocialHub 的客户分析能力带到用户已在使用的工作界面（Teams/M365 Copilot），消除系统切换成本 |
| 业务价值 | 将每企业客户内月活用户数从 3-5 人（CLI 用户）扩大到 15-30 人（含管理层、跨部门），提升平台粘性和续约率 |
| 技术目标 | 将现有 SocialHub MCP Server（stdio）扩展为 HTTP Streamable Transport，打包为可通过 Teams Admin Center 部署的标准 Teams App |

### 成功标准（6 个月）

| 指标 | 目标值 | 说明 |
|---|---|---|
| 企业月活率 | ≥ 70% | 已部署企业中，当月有活跃查询的占比 |
| 人均月查询次数 | ≥ 30 次 | 活跃用户的月均工具调用次数 |
| 多轮对话率 | ≥ 40% | 用户进行追问的会话比例，反映数据质量和用户信任度 |
| 工具覆盖率 | ≥ 5 种/用户/月 | 反映场景覆盖深度，不止局限于一种查询 |
| 集成企业续约率 vs 未集成 | 差值 ≥ 10 个百分点 | 最终业务价值验证 |
| 数据口径一致性投诉率 | < 5% | 用户反馈数字与 BI 对不上的比例，验证口径声明机制效果 |

## 二、目标用户与核心场景

### 2.1 精确用户画像

**前置准入条件（销售资格审查）**：目标企业必须满足：
1. 已为目标用户群购买 M365 Copilot 许可证（约 $30/用户/月，企业自担）
2. 目标用户日常在 Teams 中工作（非仅偶尔跨国会议）
3. 企业 IT 有 Teams 管理员或全局管理员账号

**用户画像 A：管理层（CEO/VP/总监）**

- 岗位：部门总监及以上，负责业务结果的直接汇报
- 公司规模：100-500 人的中型零售/电商企业，或跨国企业亚太区
- M365 使用现状：已有 M365 Copilot 许可证，日均在 Teams 处理 20+ 条消息
- 核心痛点：stand-up 前临时需要数字，以往依赖运营同学拉数，等待 30 分钟-2 小时
- 使用频率：每天 1-2 次，以晨会前和临时汇报前为主
- 对 CLI 态度：完全不用，不知道有此工具

**用户画像 B：运营/增长团队**

- 岗位：运营经理、增长运营、运营分析师
- 公司规模：同上
- M365 使用现状：同上，主要工作界面为 Teams + Excel/PPT
- 核心痛点：分析请求堆积，需要多个系统切换才能回答一个问题，记不住 CLI 参数
- 使用频率：每天 3-8 次，贯穿工作日
- 对 CLI 态度：部分人用，门槛高，记不住参数

**用户画像 C：客户成功/BD 经理**

- 岗位：客户成功经理（CSM），负责企业大客户续约
- 核心痛点：客户复盘电话前需要快速准备数据摘要，依赖运营同学等待 1-2 小时
- 使用频率：每周 2-3 次，以客户电话前和汇报准备时为主
- 对 CLI 态度：基本不用

**画像排除（不是目标用户）**：
- 未购买 M365 Copilot 的企业用户
- 日常工作在钉钉/企业微信而非 Teams 的用户（主要在中国大陆的非出海企业）
- 企业 IT 管理员（部署者，不是日常使用者）

### 2.2 核心用户故事（修正后版本）

**US-01：运营经理 — 每日晨会数据准备（保留，无重大修正）**

- 岗位：运营经理（中型零售企业，50 人运营团队）
- 场景：每天 9:00 stand-up 前，在 Teams 打开 SocialHub Agent，获取近 30 天关键客户健康指标
- 用户输入示例：「帮我拉一下近 30 天的客户概况，重点看留存和高价值客户数量」
- 验收标准：
  - [ ] Agent 在 5 秒内返回结构化摘要（活跃客户数、新增客户数、留存率、高价值客户数量）
  - [ ] 每个关键指标后附加数据口径说明（如「留存率 67.3%，统计口径：近 30 天有购买记录 ÷ 31-60 天有购买记录」）
  - [ ] 数据范围自动匹配「近 30 天」，并在回答开头说明所用时间范围
  - [ ] 返回内容可直接复制到 Teams 消息，格式清晰无乱码
- MCP 工具调用：`get_customer_overview` + `get_retention_analysis`

**US-02：增长团队 — 流失风险客户定位（修正：增加跳转链接）**

- 岗位：增长运营（专注用户激活和流失挽回）
- 场景：月末找出沉默高价值客户，准备挽回活动目标名单
- 用户输入示例：「找出最近 60 天没有购买、但历史消费在前 20% 的客户，大概多少人？」
- 修正说明：在返回结果末尾附加「在 SocialHub 中创建挽回活动 →」深链接，不要求 Agent 执行写操作，只提供跳转入口
- 验收标准：
  - [ ] 第一轮返回符合条件的客户数量和基本统计（含口径说明）
  - [ ] 支持追问，第二轮在第一轮基础上进一步筛选
  - [ ] 返回末尾包含 SocialHub 主产品对应功能的深链接（MVP 可以是固定 URL，EA 阶段改为带参数的深链接）
  - [ ] 多轮对话总耗时不超过 20 秒（修正：原 15 秒在串行工具调用场景下不现实）
- MCP 工具调用：`get_rfm_analysis` + `get_customer_list` + `get_retention_analysis`

**US-03：总监级管理层 — 营销活动效果快查（修正：澄清时间解析边界）**

- 岗位：市场总监（负责季度营销预算）
- 场景：路上用手机快速确认活动数据
- 修正说明：明确时间解析边界——「上周」指上一完整自然周，如果时间范围内有多个活动，Agent 返回列表让用户选择；不假设活动名称可被识别（名称需要精确匹配或时间范围足够精确）
- 验收标准：
  - [ ] 能识别「上周」等相对时间表达，自动转换日期范围，并在回答中说明
  - [ ] 若时间范围内有多个活动，返回列表供用户选择，而非猜测
  - [ ] 回答中关键数字突出显示，总长度控制在 200 字以内（适配手机竖屏）
- MCP 工具调用：`get_campaign_analysis` + `get_customer_overview`

**US-04：客户成功经理 — 大客户健康度检查（修正：澄清数据层）**

- 岗位：客户成功经理（负责 20 个企业大客户续约）
- 场景：客户复盘电话前 30 分钟准备数据摘要
- 修正说明：`get_customer_list` 工具中的「客户」指 C 端消费者聚合数据，而非 B 端企业客户档案。US-04 中的 C20231105 是该企业在 SocialHub 平台上的消费者账号或客户群标签，不是另一个企业的 ID。Instructions 中明确此工具返回消费者维度统计，若用户意图是查询某企业大客户的 B 端数据，需单独设计工具（B 端查询工具列为 EA 阶段 Backlog）。MVP 阶段 US-04 的有效性限于「某类型/某标签消费者群体的分析」，不支持「按 B 端企业账号查询」。
- 验收标准（MVP 限定范围）：
  - [ ] 支持传入消费者筛选条件（如最近 90 天、高价值群体），返回该群体的聚合分析数据
  - [ ] 如发现风险信号（LTV 下滑、沉默周期延长），Agent 主动标注 ⚠️
  - [ ] 输出格式可方便复制到邮件摘要
- MCP 工具调用：`get_ltv_analysis` + `get_customer_list` + `get_anomaly_detection`

**US-05：运营团队 — 异常预警主动发现（保留，无重大修正）**

- 岗位：运营分析师（负责日常数据监控）
- 场景：周一上午主动检查上周是否有数据异常
- 用户输入示例：「上周有什么数据异常吗？重点看留存和订单」
- 验收标准：
  - [ ] Agent 主动扫描异常，无需用户指定具体指标
  - [ ] 如无异常，明确告知「上周主要指标正常，无显著异常」
  - [ ] 如有异常，按严重程度排序列出，每条附带口径说明和建议行动（含深链接）
- MCP 工具调用：`get_anomaly_detection` + `get_order_trends` + `get_retention_analysis`

### 2.3 使用频率假设

| 岗位 | 日均会话次数 | 平均工具调用次/会话 | 月均工具调用次 |
|---|---|---|---|
| 管理层 | 1-2 次 | 1.5 个工具 | 约 45-90 次调用 |
| 运营/增长 | 3-8 次 | 2 个工具 | 约 90-480 次调用 |
| 客户成功/BD | 0.5-1 次（约 2-3 次/周） | 2 个工具 | 约 30-60 次调用 |

基于以上假设，每企业（10 名活跃用户）月均工具调用量约 300-600 次，Render Starter 层（并发限制约 100 req/min）完全可以承载。

## 三、功能规格

### 3.1 MCP 工具清单（8 个核心工具）

以下 8 个工具是 MVP 阶段唯一暴露给 M365 Copilot 的工具（写入 mcp-tools.json）。MCP Server 中的其余 28+ 工具仍然保留但不对 M365 Copilot 公开，避免 token 预算超限和工具路由混乱。

---

**工具 1：get_customer_overview**

- 描述：获取企业客户整体健康状况快照，返回活跃客户数量、新增客户数、整体留存率、高价值客户占比等核心摘要指标。当用户询问「整体情况」「客户概况」「关键数字」「数据总结」时调用此工具。适用于管理层晨会快查和运营每日例行监控。
- 必填参数：`tenant_id`（字符串，由认证层注入，客户端不传）、`days`（整数，默认 30，表示近 N 天）
- 可选参数：`start_date`（字符串，YYYY-MM-DD）、`end_date`（字符串，YYYY-MM-DD）
- 响应格式：包含 active_customers、new_customers、retention_rate、high_value_ratio、data_range_desc（口径说明）字段的 JSON，由 Agent 格式化为中文摘要

---

**工具 2：get_retention_analysis**

- 描述：分析客户留存率和流失趋势，返回指定周期内的留存率、流失率、复购间隔分布。当用户询问「流失」「留存」「复购」「回头客」「客户流走了多少」时调用此工具。留存率口径：有 N+1 周期购买的用户 ÷ 有 N 周期购买的用户。
- 必填参数：`tenant_id`（认证层注入）、`days`（默认 30）
- 可选参数：`comparison_period`（对比周期天数，默认与 days 相同）
- 响应格式：包含 retention_rate、churn_rate、period_definition（口径说明）、trend_vs_previous 的 JSON

---

**工具 3：get_rfm_analysis**

- 描述：按 RFM 模型（最近购买时间/购买频率/消费金额）对客户进行分层，返回各层级的客户数量和占比。当用户询问「高价值客户」「VIP」「客户分层」「哪类客户最好」「客户细分」时调用。RFM 分层阈值基于该企业历史数据的分位数动态计算。
- 必填参数：`tenant_id`（认证层注入）
- 可选参数：`days`（分析时间窗口，默认 90）、`segments`（分层数，默认 5）
- 响应格式：包含各层级（冠军客户/忠实客户/潜力客户/流失风险/流失客户）的 customer_count、percentage、avg_ltv，以及 segment_method（口径：阈值计算方式）字段

---

**工具 4：get_customer_list**

- 描述：按条件筛选并返回符合条件的消费者统计列表，返回该群体的规模、平均消费、沉默天数等聚合数据（不返回个人姓名、手机号等 PII 字段）。当用户需要「找出哪些客户」「多少人满足条件」「给我一个名单规模」时调用。
- 必填参数：`tenant_id`（认证层注入）
- 可选参数：`min_days_since_last_purchase`（最近购买距今最少天数）、`max_days_since_last_purchase`（最多天数）、`ltv_percentile_min`（LTV 分位数下限，0-100）、`days`（历史统计窗口，默认 90）、`limit`（返回条数上限，默认 50）
- 响应格式：包含 total_count、sample_stats（均值/中位数/区间）、filter_applied（查询条件回显，作为口径说明）的 JSON

---

**工具 5：get_anomaly_detection**

- 描述：对多维度指标进行异常检测，返回当前周期内相对历史基线有显著偏差的指标列表，按严重程度排序。当用户询问「有没有问题」「需要关注什么」「有什么异常」「上周出了什么事」时调用。异常判断基准：相比过去 4 个同期周期的均值偏差超过 2 个标准差。
- 必填参数：`tenant_id`（认证层注入）、`days`（检查窗口，默认 7，通常检查「上周」）
- 可选参数：`dimensions`（检查维度，默认 all，可指定 retention/orders/campaigns）
- 响应格式：包含 anomalies 数组（每条含 metric、severity、value、baseline、deviation_pct、anomaly_definition 字段），以及 summary（如无异常时的说明）

---

**工具 6：get_order_trends**

- 描述：返回指定时间范围内的订单量、GMV 趋势，以及与上一同等周期的对比变化。当用户询问「业绩」「销售额」「订单量」「趋势」「和上月相比」时调用。与 get_customer_overview 互补：trends 提供时间轴趋势，overview 提供当前截面数据。
- 必填参数：`tenant_id`（认证层注入）、`days`（默认 30）
- 可选参数：`granularity`（日/周/月，默认 day）、`compare_previous`（布尔值，是否附加同期对比，默认 true）
- 响应格式：包含 period_series（时间序列数组）、total_gmv、total_orders、vs_previous_pct（同期对比百分比变化）、period_definition（口径：时间段定义）

---

**工具 7：get_ltv_analysis**

- 描述：分析客户生命周期价值（LTV）分布，返回高价值客户（LTV 前 20%）的规模、活跃度趋势及流失风险指标。当用户询问「LTV」「客户价值」「高潜力客户」「续费价值」「值钱的客户」时调用。LTV 计算基于历史消费总额（不含退款）。
- 必填参数：`tenant_id`（认证层注入）
- 可选参数：`top_percentile`（关注的高价值客户分位数，默认 20）、`days`（分析窗口，默认 180）
- 响应格式：包含 top_segment_count、top_segment_avg_ltv、activity_trend（最近是否活跃）、churn_risk_count（高价值客户中有流失风险的数量）、ltv_definition（口径说明）

---

**工具 8：get_campaign_analysis**

- 描述：返回指定时间范围内营销活动的参与人数、新客转化数、复购激活数、GMV 贡献及与历史活动的对比。当用户询问「活动」「大促」「营销」「转化率」「活动效果」时调用。若时间范围内有多个活动，返回列表供用户选择或分别展示。
- 必填参数：`tenant_id`（认证层注入）、`days`（活动所在时间范围，默认 14）
- 可选参数：`campaign_name`（活动名称关键词，模糊匹配）、`start_date` / `end_date`
- 响应格式：包含 campaigns 数组（每条含 name、participants、new_customers、re-engaged、gmv、vs_last_campaign_pct）、date_range_used（口径：实际使用的日期范围）

**3.2.1 完整 Instructions 文本（修正后版本）**：

```
你是 SocialHub 客户智能助手，帮助企业运营和管理团队快速获取客户数据洞察。

## 你的能力范围
你只分析 SocialHub 系统中该企业的客户数据，包括：客户概况、留存分析、RFM 分层、订单趋势、异常检测、营销活动效果、客户生命周期价值。

## 调用工具的规则
1. 当用户的问题涉及「整体情况/概况/摘要/关键数字」时，优先调用 get_customer_overview
2. 当用户问到「流失/留存/复购/回头客」时，调用 get_retention_analysis
3. 当用户问到「高价值客户/大客户/VIP/客户分层/RFM/值钱的客户」时，调用 get_rfm_analysis
4. 当用户需要「具体客户群规模/找出哪些客户/多少人满足条件」时，调用 get_customer_list
5. 当用户问「有没有问题/有什么异常/需要关注什么/上周出了什么」时，调用 get_anomaly_detection
6. 当用户问到「业绩/订单量/销售额/趋势/和上月比」时，调用 get_order_trends
7. 当用户问到「LTV/客户价值/高潜力/续费价值/值钱」时，调用 get_ltv_analysis
8. 当用户问到「活动/营销/大促/转化率/活动效果/双十一」时，调用 get_campaign_analysis
9. 如果一个问题同时涉及多个维度，可以顺序调用 2-3 个工具，综合结果后统一回答
10. 如果用户的问题超出上述工具的覆盖范围，明确告知「当前版本暂不支持查询 [具体内容]」，不要编造数据

## 数据口径透明化（重要）
每次返回关键指标时，必须附加一行口径说明。格式示例：
- 「留存率 67.3%（统计口径：近 30 天有购买记录的用户 ÷ 31-60 天内有购买记录的用户）」
- 「高价值客户 1,234 人（口径：历史消费金额位于前 20%，基于近 180 天数据）」
- 「异常判断基准：相比过去 4 周均值偏差超过 2 个标准差」
口径说明使用小号文字或放在括号内，不占用主要回答篇幅。

## 时间范围处理
- 用户说「最近」「近期」时，默认使用近 30 天
- 用户说「上周」时，使用上一个完整自然周（周一至周日）
- 用户说「本月」时，使用当月 1 日至今
- 在回答的第一行说明实际使用的时间范围，例如「以下是 2026-02-28 至 2026-03-29 的数据：」
- 时间表述模糊时，先给出基于默认假设的回答，末尾注明「如需调整时间范围，请告诉我」
```

### 3.4 认证方案规格

#### MVP 方案：API Key via ATK（M365 Agents Toolkit）

**适用阶段**：内测（Phase 0）+ EA（Phase 1-2）

**实现方式**：
- API Key 由 SocialHub 运营生成，通过 SocialHub 客户后台安全链接发送给企业 IT 管理员
- IT 管理员在 M365 Agents Toolkit 的 `env/.env.production` 中配置一次，员工不可见
- 服务端维护 `API_KEY → tenant_id` 映射表，客户端禁止传入 tenant_id 参数
- MCP Server 的 `auth.py` 中间件：从请求 Header 提取 `X-API-Key`，查映射表注入 `tenant_id` 到请求上下文

**API Key 安全规范**：
- 格式：`sh_` 前缀 + 32 字节随机字符串（Base64URL 编码，共约 48 字符）
- 传输：仅通过 SocialHub 客户后台安全链接发送，禁止明文邮件
- 轮换周期：每 6 个月例行轮换，或企业反馈 Key 可能泄露时立即轮换
- 轮换流程：企业 IT 管理员在 SocialHub 客户后台自助申请新 Key，旧 Key 立即失效（MVP 阶段此自助界面为 P2，过渡方案为邮件申请 SocialHub 运营处理，SLA：4 小时响应）

**plugin.json auth 配置（MVP）**：
```json
{
  "auth": {
    "type": "ApiKeyPluginVault",
    "reference_id": "<ATK_env_reference>"
  }
}
```

#### GA 方案：Entra ID OAuth（OAuthPluginVault）

**适用阶段**：EA 结束前完成，GA 上线必须

**切换触发条件**：达到 5 个 EA 客户且月活率 ≥ 70%，或任一客户提出合同级安全要求

**实现方式**：
- Azure AD 应用注册，配置 redirect URI 为 M365 Teams 回调地址
- OAuth Scope：仅 `openid profile`（不申请 Graph API 权限，最小化安全面）
- Teams Developer Portal 注册 OAuth 配置，获取 reference_id
- MCP Server `auth.py` 增加 Entra token 验证路径：验证签名 → 提取 `tid` claim → 映射 SocialHub tenant_id
- plugin.json 切换 auth.type 为 `OAuthPluginVault`

**员工离职自动失权**：Entra 账号禁用后，下次 token 刷新时 OAuth token 自动失效，MCP Server 拒绝请求，SocialHub 侧无需额外操作。

## 四、数据准确性保证

（直接回应客户审视致命问题 2 和缺失 2）

### 4.1 数据口径声明设计

**设计原则**：每次返回关键指标时，Agent 必须在同一回答内附加该指标的计算口径。口径信息来源于 MCP 工具的响应 JSON 中专门的 `*_definition` 字段，由工具实现者维护，不由 Agent 自行编写（防止 LLM 编造口径）。

**工具响应中的口径字段规范**：

| 工具 | 口径字段 | 示例值 |
|---|---|---|
| `get_retention_analysis` | `period_definition` | 「近 30 天有购买记录的用户数 ÷ 31-60 天内有购买记录的用户数」 |
| `get_rfm_analysis` | `segment_method` | 「基于近 90 天数据，按消费金额/频率/最近购买日分位数动态计算，5 个层级」 |
| `get_customer_overview` | `active_definition` | 「活跃客户：近 30 天内有至少 1 次购买记录」 |
| `get_anomaly_detection` | `anomaly_definition` | 「与过去 4 个同期均值偏差超过 2 个标准差」 |
| `get_ltv_analysis` | `ltv_definition` | 「历史消费总额（不含退款），统计截止今日」 |
| 所有工具 | `data_range_desc` | 「统计范围：2026-02-28 至 2026-03-29」 |

**Instructions 对口径输出的约束**：已写入 3.2.1 完整 Instructions 文本「数据口径透明化」一节，要求 Agent 必须在每次回答关键指标时输出对应的口径说明。

### 4.2 引用来源展示机制（Copilot Attribution）

M365 Copilot 原生支持 attribution 功能：当工具调用结果作为 Copilot 回答的依据时，Copilot 可以在回答末尾展示「来源：SocialHub 客户数据（通过 SocialHub 客户智能助手）」的引用标注。

**实现方式**：MCP 工具的响应 TextContent 中包含 `source` 字段，M365 Copilot 运行时识别后在 UI 中渲染为可点击的来源标注。

**用户价值**：用户在质疑数字时，可以点击来源标注，确认数字来自 SocialHub 数据系统而非 LLM 编造，建立对数字的基础信任。

## 五、可观测性最低要求

（直接回应客户审视致命问题 3 和缺失 1）

### 5.1 /health 端点规范

**端点路径**：`GET /health`

**无认证要求**：/health 端点不要求 API Key 或 OAuth token，可公开访问（不返回敏感数据）

**响应规范**：

```json
// 正常状态 (HTTP 200)
{
  "status": "ok",
  "version": "1.2.0",
  "timestamp": "2026-03-29T10:00:00Z",
  "checks": {
    "mcp_server": "ok",
    "upstream_connection": "ok",
    "cache": "ok"
  }
}

// 降级状态 (HTTP 200，但 status 为 degraded)
{
  "status": "degraded",
  "version": "1.2.0",
  "timestamp": "2026-03-29T10:00:00Z",
  "checks": {
    "mcp_server": "ok",
    "upstream_connection": "error",
    "cache": "ok"
  },
  "message": "上游数据连接异常，工具调用可能返回缓存数据或失败"
}

// 故障状态 (HTTP 503)
{
  "status": "down",
  "timestamp": "2026-03-29T10:00:00Z",
  "message": "服务不可用"
}
```

**探测配置（Render）**：Render 健康检查每 5 分钟探测一次 /health，连续 3 次失败（15 分钟）触发自动重启。

**面向 IT 管理员的状态页**：Render Starter 层提供内置状态监控页，SocialHub 将此 URL 写入部署文档：`https://status.render.com`（或申请 `status.socialhub.ai` CNAME 到 Render 状态页），作为 IT 管理员和企业高层的服务健康自查入口。

### 5.2 错误响应规范

MCP Server 的所有工具处理器在出错时，必须返回 `list[TextContent]`（不抛出未捕获异常），错误消息必须对用户友好，不暴露技术细节（堆栈、SQL 错误、内部 ID 等）。

**错误类型映射**：

| 错误来源 | HTTP 状态 | 返回给 Agent 的 TextContent | Agent 展示给用户的文本 |
|---|---|---|---|
| 查询结果为空（正常业务情况） | 200 | `{"result": "empty", "suggestion": "..."}` | 「该时间段内暂无符合条件的数据。建议调整时间范围或放宽筛选条件。」 |
| 上游数据查询超时（> 10 秒） | 200 | `{"error": "timeout", "retry_suggested": true}` | 「数据查询耗时较长，请稍后重试。如持续出现，请联系 IT 管理员。」 |
| tenant_id 未找到（新客户未配置映射） | 200 | `{"error": "tenant_not_found"}` | 「未能找到贵司的数据配置。请联系 IT 管理员确认 SocialHub 服务是否已开通。」 |
| API Key 无效/过期 | 401 | （M365 Copilot 层拦截） | 「身份验证失败，请联系 IT 管理员重新配置访问凭证。」 |
| MCP Server 内部错误 | 500 | `{"error": "internal_error", "reference_id": "<uuid>"}` | 「服务遇到意外错误（错误编号：<reference_id>）。请截图此信息发送给 IT 管理员或 SocialHub 支持。」 |
| 查询超出工具能力范围 | 200 | `{"error": "out_of_scope", "scope_description": "..."}` | 「当前版本暂不支持查询 [具体内容]。该功能在后续版本中计划支持。」 |

**故障通知流程（运营 SOP，不需要工程开发）**：

1. Render 健康检查连续 3 次失败 → 触发 Render 告警邮件到 SocialHub 值班运营邮箱
2. SocialHub 运营值班接到告警后：
   - 30 分钟内：向所有已登记 IT 联系人邮箱发送「服务故障通知」（模板预备好，填写预计恢复时间即可发送）
   - 同时更新 `status.socialhub.ai` 状态页（Render 状态页可自动更新，或手动更新文字说明）
   - 1 小时内：通过 SocialHub 运营 Teams 账号向企业 IT 联系人发送 Teams 消息（双渠道通知）
3. 故障恢复后：向 IT 联系人发送「服务恢复通知」
4. 故障后 24 小时内：提供故障原因摘要和预防措施说明

## 九、附录：对抗回应记录

以下是对客户审视（04-customer-review.md）中提出的所有问题的逐条回应。

### 致命问题

**致命问题 1：查询结果无法直接触发行动，分析价值大打折扣**

**决策：部分接受**

接受的部分：确认「分析结果的可操作性」是真实的用户痛点，当前方案只解决了「看数据」环节，没有覆盖「执行动作」环节，导致部分运营经理的工作路径没有实质缩短。

修改方式（已纳入 PRD）：在 MCP 工具的响应 JSON 中增加 `action_link` 字段，返回 SocialHub 主产品对应功能页面的固定深链接（MVP：固定 URL；EA 阶段：带预填参数的深链接）。Instructions 要求 Agent 在回答末尾展示跳转入口。这个修改无需工程开发，只需工具响应 JSON 结构调整和 Instructions 一行配置。

拒绝的部分：不在 MVP 阶段实现 Agent 直接执行写操作（创建活动、发送优惠券）。理由：
1. 范围边界：写操作需要 SocialHub 主产品 API 的写权限设计，超出当前 MCP Server 只读分析的设计范围
2. 安全风险：通过自然语言 Agent 触发批量营销动作（如「给这批客户发优惠券」）会引入误操作风险，需要更复杂的确认机制
3. 客户审视中的核心论点是「整体效率没有本质提升」——深链接虽然不消除系统切换，但将切换成本从「找入口」降低为「一键跳转」，阶段性地改善了可操作性

**致命问题 2：数据准确性验证机制缺失，企业用户无法建立信任**

**决策：完全接受**

修改方式（已纳入 PRD）：
1. 所有工具响应 JSON 增加 `*_definition` 和 `data_range_desc` 字段（见 4.1 数据口径声明设计）
2. Instructions 增加「数据口径透明化」强制要求（见 3.2.1）
3. 利用 M365 Copilot attribution 能力展示数据来源（见 4.2）

理由：这个修改工程成本接近零（工具响应 JSON 加字段 + Instructions 增加规则），但对建立数据信任至关重要。一次「数字对不上」带来的信任损耗需要十次「数字对上」才能修复，主动透明化口径的边际收益极高。

**致命问题 3：IT 管理员的故障感知和排障路径几乎不存在**

**决策：接受主干，部分延迟**

接受并纳入 MVP 的部分：
1. /health 端点规范（见 5.1）——工程成本约 2 小时
2. 错误响应规范（见 5.2）——工程成本约 0.5 天
3. 状态页 URL 写入部署文档（使用 Render 自带状态页，零工程成本）
4. 故障通知 SOP（运营流程文档，零工程成本）

延迟到 EA 阶段的部分：独立的 `status.socialhub.ai` 自建状态页。理由：Render 自带状态页已满足 IT 采购合规的基本要求（有公开可访问的服务状态 URL）；自建状态页（如使用 Betterstack）需要额外订阅成本和集成工作量，延迟到 EA 阶段（客户数量 5-10 家时）更合理。

对「30 分钟内通知」的运营 SOP 化处置：客户审视指出「通知渠道是什么，由谁发，这不是可靠的 SLA 承诺」——完全正确。修正方式已在 5.2 故障通知流程（运营 SOP）中明确：Render 告警邮件 → SocialHub 值班运营邮件 + Teams 双渠道通知 IT 联系人。这是一个 SOP 文档设计，不需要工程开发，但必须在首批客户部署前完成。

---

### 重要偏差

**偏差 1：「节省切换时间」被高估，实际摩擦被低估（习惯养成问题）**

**决策：部分接受，延迟解决**

接受：方案确实对习惯建立的冷启动成本估计不足，「零切换成本」是稳态假设而非初始状态。

不在 MVP 修改的理由：习惯养成机制（使用提醒、主动推送）超出 Declarative Agent 的能力边界（不支持主动推送），需要 Bot 架构。MVP 阶段的缓解措施：
- 管理员配置部署范围时，向目标员工发送一封「你可以这样用 SocialHub Copilot Agent」的 Teams 消息或邮件（运营流程，非产品功能）
- Conversation Starters 覆盖所有主要岗位的高频痛点，降低第一次使用的认知成本

**偏差 2：管理层使用场景（US-03）过于理想化**

**决策：部分接受**

接受：已在 US-03 修正版中处理「时间范围内多个活动时返回列表」的边界情况。

对移动端验证的处理：延迟到 EA 阶段专项测试（见 MVP 排除列表），不阻塞 MVP 发布，但列入 EA 验收标准。

**偏差 3：US-04 场景的 B 端/C 端数据层矛盾**

**决策：完全接受**

修改方式（已纳入 PRD）：US-04 已在 2.2 中重新限定为「C 端消费者聚合分析」，明确 `get_customer_list` 工具不支持 B 端企业账号查询。B 端查询工具列为 EA 阶段 Backlog。

**偏差 4：度量体系是 SocialHub 视角，不是客户视角**

**决策：部分接受**

接受：成功标准（一节）已增加「数据口径一致性投诉率」指标，间接反映客户信任度。内测阶段的业务验收标准 B3 也是客户视角指标。

拒绝的部分：「因为 Agent 发现异常而避免的客户流失数量」这类业务结果指标在 MVP 阶段无法可靠度量（需要实验组/对照组设计），作为长期 OKR 指标在 GA 阶段建立度量体系更合适。

### 缺失

**缺失 1：面向企业 IT 的服务状态透明化机制**

**决策：接受，MVP 实现最简版本**

修改方式：Render 状态页 URL 写入部署文档 + /health 端点公开可访问（见 5.1）。自建状态页延迟到 EA 阶段。

**缺失 2：数据口径说明和计算逻辑文档**

**决策：完全接受，进入 MVP**

修改方式：见 4.1 数据口径声明设计，工具响应 JSON 增加口径字段，Instructions 要求 Agent 输出口径说明。

**缺失 3：API Key 轮换和吊销的自助流程**

**决策：MVP 接受最简版本，GA 前完成自助界面**

MVP 过渡方案：企业 IT 通过邮件申请 SocialHub 运营处理 Key 轮换，SLA 4 小时响应。这不符合「合规要求的 N 分钟内吊销」，但在 EA 阶段（5-10 个客户，风险可控）可接受。

不接受立即实现自助界面的理由：自助 Key 管理界面需要 SocialHub 主产品后台开发工作（独立于本 PRD），EA 阶段人工流程过渡是合理的成本取舍。GA 前必须完成。

**缺失 4：使用报告（Usage Report）给管理层**

**决策：MVP 接受运营手动版本，GA 前实现自动化**

MVP 方案：SocialHub 运营每月手动导出工具调用统计，发送给已登记联系人邮箱（约 30 分钟/月/客户，EA 阶段 10 个客户约 5 小时/月，可接受）。GA 阶段实现自动化月报。

**缺失 5：出错时的用户引导（Error UX）**

**决策：完全接受，进入 MVP**

修改方式：见 5.2 错误响应规范，已区分「无数据」「超时」「认证失败」「内部错误」「超出范围」五种情况，每种提供不同的用户引导文字。

**缺失 6：多语言/多时区支持路线图**

**决策：给出路线图，不纳入 MVP**

路线图：
- MVP：中文唯一语言，Instructions 中「除非用户明确要求英文」留有切换口
- EA 阶段（视内测数据决定）：英文 Conversation Starters + 英文 Instructions 并行版本
- GA：根据客户分布决定是否需要多语言 Agent 包（可能是独立的英文版 ZIP 包）

不纳入 MVP 的理由：首批内测客户为中文工作环境的出海企业；英文适配需要工具描述和 Instructions 的完整双语化，工作量约 0.5-1 天，但在首批客户验证之前花这个成本不合理。
