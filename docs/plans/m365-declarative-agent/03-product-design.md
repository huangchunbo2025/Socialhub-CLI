## 一、用户故事与核心场景

### 用户故事设计原则

每个故事遵循「岗位 + 场景 + 验收标准 + 工具调用」四要素。工具调用列表来自 MCP Server 的 36+ 工具中精选的核心集，每个故事只调用 1-3 个工具，保证单次对话的响应质量。

---

### US-01：运营经理 — 每日晨会数据准备

**岗位**：运营经理（中型零售企业，50 人运营团队）

**场景**：每天 9:00 stand-up 前，在 Teams 聊天中打开 SocialHub Agent，用一句话获取昨日和近 30 天的关键客户健康指标，直接粘贴到晨会的 Teams 频道里。

**用户输入示例**：
> 「帮我拉一下近 30 天的客户概况，重点看留存和高价值客户数量」

**验收标准**：
- [ ] Agent 在 5 秒内返回结构化摘要，包含：活跃客户数、新增客户数、留存率、高价值客户数量
- [ ] 数据范围自动匹配「近 30 天」，无需用户手动指定日期
- [ ] 返回内容可直接复制到 Teams 消息，格式清晰（无乱码、无 JSON 裸数据）
- [ ] 运营经理无需切换到任何其他系统

**MCP 工具调用**：
1. `get_customer_overview`（主要：获取整体客户数、活跃数、新增数）
2. `get_retention_analysis`（辅助：获取留存率趋势）

---

### US-02：增长团队 — 流失风险客户定位

**岗位**：增长运营（专注用户激活和流失挽回）

**场景**：月末复盘时，需要快速找出上个月消费过但本月完全沉默的高价值客户，准备定向挽回活动的目标名单。

**用户输入示例**：
> 「找出最近 60 天没有购买、但历史消费在前 20% 的客户，大概多少人？」

**追问示例**：
> 「这些客户里有多少人上次购买是 30-60 天前？」

**验收标准**：
- [ ] 第一轮对话返回符合条件的客户数量和基本统计（金额区间、平均沉默天数）
- [ ] 支持追问，第二轮对话能在第一轮结果基础上进一步筛选，无需重复描述背景
- [ ] 返回数据中包含可操作的下一步建议（如「建议针对该群体发送 X 类型优惠券」）
- [ ] 多轮对话总耗时不超过 15 秒

**MCP 工具调用**：
1. `get_rfm_analysis`（识别高价值客群的 R/F/M 分布）
2. `get_customer_list`（筛选符合条件的具体客户列表）
3. `get_retention_analysis`（辅助验证流失时间分布）

---

### US-03：总监级管理层 — 营销活动效果快查

**岗位**：市场总监（负责季度营销预算）

**场景**：刚开完供应商会议，在回办公室的路上用手机打开 Teams，想快速确认上周双十一大促活动的实际转化数据是否达到预期，不需要等运营同学整理 PPT。

**用户输入示例**：
> 「上周那个双十一活动最终带来多少新客户，复购用户有多少？」

**验收标准**：
- [ ] 能够识别「上周」等相对时间表达，自动转换为具体日期范围
- [ ] 返回内容包含：活动参与人数、新客转化数、复购激活数、对比上次活动的变化趋势
- [ ] 内容在手机屏幕（竖屏）上可读，关键数字突出显示
- [ ] 不需要用户记得具体活动名称或 ID，能通过时间范围定位活动

**MCP 工具调用**：
1. `get_campaign_analysis`（活动参与、转化、GMV 数据）
2. `get_customer_overview`（新客占比验证）

---

### US-04：客户成功经理 — 大客户健康度检查

**岗位**：客户成功经理（负责 20 个企业大客户的续约）

**场景**：下午有一个企业客户的季度复盘电话，需要在 30 分钟内准备好该客户近 90 天的业务数据摘要，过去需要找运营同学拉数并等待 1-2 小时。

**用户输入示例**：
> 「帮我看一下这家客户近 90 天的购买情况和复购趋势，他们用的账号 ID 是 C20231105」

**验收标准**：
- [ ] 能够接受客户 ID 作为参数，返回该客户维度的分析数据（而非全局汇总）
- [ ] 包含近 90 天的订单数、GMV、复购次数、与前期对比的趋势变化
- [ ] 如果发现该客户有风险信号（如 LTV 下滑、最近沉默），Agent 主动标注
- [ ] 输出格式可以方便地复制到邮件或 PPT 摘要框中

**MCP 工具调用**：
1. `get_customer_list`（获取指定客户的基本信息和订单摘要）
2. `get_ltv_analysis`（该客户的生命周期价值和趋势）
3. `get_anomaly_detection`（是否存在异常信号）

---

### US-05：运营团队 — 异常预警主动发现

**岗位**：运营分析师（负责日常数据监控）

**场景**：周一上午，想主动检查上周是否有任何数据异常——不是回答具体问题，而是「有什么需要我关注的问题」，相当于把 Agent 当作智能监控告警使用。

**用户输入示例**：
> 「上周有什么数据异常吗？重点看留存和订单」

**验收标准**：
- [ ] Agent 能够主动扫描异常，而不是等待用户指定具体指标
- [ ] 如无异常，明确告知「上周主要指标正常，无显著异常」
- [ ] 如有异常，按严重程度排序列出，每条异常附带简短解释和建议动作
- [ ] 对于严重异常（如某门店订单量暴跌 50%），结果语气明显区分于轻微波动

**MCP 工具调用**：
1. `get_anomaly_detection`（主要：多维度异常扫描）
2. `get_order_trends`（辅助：订单维度趋势验证）
3. `get_retention_analysis`（辅助：留存维度验证）

---

### US-06（扩展场景）：连锁零售 — 门店维度分析

**岗位**：大区运营经理（管理 50 家线下门店）

**场景**：月度区域会议前，需要快速找出本月表现最差的 5 家门店及其主要问题（是客单价低、客流量少还是复购差），作为会议议题的数据支撑。

**用户输入示例**：
> 「本月哪几家门店表现最差？分别差在哪个维度上？」

**验收标准**：
- [ ] 返回按综合得分排名的门店列表，每家标注核心问题维度
- [ ] 支持追问「某家门店上个月表现怎样」，进行月度对比
- [ ] 数据颗粒度：门店级别，不仅仅是全局汇总

**MCP 工具调用**：
1. `get_store_analysis`（门店维度的客流、GMV、复购数据）
2. `get_anomaly_detection`（门店级别的异常识别）

## 二、功能设计

### 2.1 Declarative Agent 能力边界

#### 能做什么

| 能力 | 说明 |
|---|---|
| **自然语言查询数据** | 将口语化问题映射到 MCP 工具调用，返回结构化摘要 |
| **多轮追问** | 在同一对话线程内保持上下文，支持「这些客户里……」「再看一下……」等追问 |
| **相对时间解析** | 「上周」「近 30 天」「本月」等自动换算为具体日期范围 |
| **主动异常发现** | 在用户不指定具体问题时，扫描多维度异常并排序呈现 |
| **数据摘要格式化** | 以适合复制粘贴到 Teams 消息/邮件的格式输出，关键数字加粗 |
| **租户隔离查询** | 用户只能查询本公司数据，由 Entra ID token 中的租户 ID 强制限定 |
| **工具路由自主决策** | Agent 根据问题语义自主选择调用哪些工具，用户无需知道工具名称 |

#### 不能做什么（明确边界）

| 限制 | 说明 | 替代路径 |
|---|---|---|
| **不能修改数据** | Agent 是只读分析，不能创建活动、修改客户信息、触发营销动作 | 通过 SocialHub 主产品操作 |
| **不能实时推送** | 不主动向用户推送告警，只响应用户主动查询 | 后续迭代：定时摘要 Workflow |
| **不能跨租户查询** | 严格禁止，任何跨租户数据访问在认证层阻断 | 技术约束，不可配置 |
| **不能查询个人隐私数据** | 只返回聚合统计和脱敏摘要，不展示具体消费者个人信息（姓名、手机号等） | 合规约束 |
| **不能操作 M365 数据** | 不读写 SharePoint、Outlook、Teams 频道中的内容 | Agent 范围限定在 SocialHub 数据 |
| **不能处理图片/附件** | 纯文本问答，不支持上传截图或分析文件 | 后续迭代 |
| **工具描述之外的查询** | 如果问题超出 8 个核心工具的覆盖范围，Agent 会明确告知「暂不支持」而不是编造数据 | 后续迭代扩展工具集 |

---

### 2.2 精选 8 个核心 MCP 工具（从 36+ 中筛选）

筛选标准（优先级从高到低）：
1. 查询频率：每周被触发概率 > 50%
2. 独立可回答：单个工具即可完整回答一类问题，不依赖其他工具的输出
3. 非技术用户友好：返回结果可直接理解，不需要解码
4. 业务差异化：体现 SocialHub 核心能力（留存/RFM/LTV），不是通用 BI 工具能回答的

| 优先级 | 工具名 | 核心场景 | 筛选理由 |
|---|---|---|---|
| **P0 必选** | `get_customer_overview` | 每日晨会、管理层快查 | 频率最高，是所有其他查询的入口和背景参照 |
| **P0 必选** | `get_retention_analysis` | 流失预警、留存趋势 | SocialHub 核心差异化能力，是客户续约 SocialHub 的主要理由 |
| **P0 必选** | `get_rfm_analysis` | 客户分层、高价值识别 | 运营决策的基础框架，几乎所有细分场景都依赖 RFM 分层 |
| **P0 必选** | `get_customer_list` | 钻取明细、客成场景 | 从汇总到明细的关键桥梁，不提供此能力会导致「只能看总数不能行动」的无力感 |
| **P1 核心** | `get_anomaly_detection` | 主动监控、风险发现 | 唯一能回答「有什么问题」而非「这个指标怎么样」的工具，体现智能化价值 |
| **P1 核心** | `get_order_trends` | 业绩趋势、活动效果 | 管理层和运营最关心的业绩维度，与 overview 互补（趋势 vs 截面） |
| **P1 核心** | `get_ltv_analysis` | 高价值客户、续费判断 | 客成场景的核心工具，LTV 是续约谈判的重要依据 |
| **P1 核心** | `get_campaign_analysis` | 营销复盘、活动 ROI | 增长团队高频需求，每次活动结束后必查 |

**排除说明**（有意不选）：
- `get_product_analysis` / `get_store_analysis`：行业特化工具，仅对连锁零售客户有价值，MVP 阶段通过第二个 Plugin 扩展（P2 预留）
- `get_points_analysis` / `get_coupon_analysis` / `get_loyalty_analysis`：运营工具，场景频率低于上述 8 个，且依赖其他工具结果才有完整意义
- `get_repurchase_analysis`：功能已由 `get_retention_analysis` + `get_rfm_analysis` 组合覆盖，避免工具路由混乱

**Token 预算说明**：8 个工具的 Schema 总 token 占用约 2,000-3,000 tokens，在 M365 Copilot 的工具上下文窗口（通常 10,000 tokens）内留有充足余量，确保工具路由质量。

### 2.3 对话提示词（Conversation Starters）设计

设计原则：
- 每个 Starter 对应一个真实岗位的高频痛点，不是功能演示
- 措辞口语化，降低用户的认知门槛
- 6 个 Starter 覆盖 3 个岗位（管理层 2 个、运营 2 个、客成 2 个）
- 每个 Starter 都能被直接点击并立即获得有价值的回答

| # | Starter 文案 | 目标岗位 | 触发工具 |
|---|---|---|---|
| 1 | 近 30 天的客户健康度怎么样？给我一个关键数字的快速摘要 | 管理层 | `get_customer_overview` + `get_retention_analysis` |
| 2 | 上周有没有数据异常？重点看订单量和客户留存 | 运营分析师 | `get_anomaly_detection` + `get_order_trends` |
| 3 | 帮我找出最近 60 天没有复购、但历史消费金额较高的客户群 | 增长运营 | `get_rfm_analysis` + `get_customer_list` |
| 4 | 最近一次营销活动的效果怎么样？新客和复购各带来了多少？ | 市场/增长 | `get_campaign_analysis` |
| 5 | 我们的高价值客户（LTV 前 20%）最近活跃度如何？有流失风险吗？ | 客户成功 | `get_ltv_analysis` + `get_retention_analysis` |
| 6 | 本月整体业绩趋势如何？和上个月相比有什么变化？ | 管理层 | `get_order_trends` + `get_customer_overview` |

---

### 2.4 Agent Instructions 设计

Agent Instructions 是 `declarativeAgent.json` 中 `instructions` 字段的内容，直接决定 Copilot 的行为模式和输出质量。

#### 完整 Instructions 文本

```
你是 SocialHub 客户智能助手，帮助企业运营和管理团队快速获取客户数据洞察。

## 你的能力范围
你只分析 SocialHub 系统中该企业的客户数据，包括：客户概况、留存分析、RFM 分层、订单趋势、异常检测、营销活动效果、客户生命周期价值。

## 调用工具的规则
1. 当用户的问题涉及「整体情况/概况/摘要」时，优先调用 get_customer_overview
2. 当用户问到「流失/留存/复购/回头客」时，调用 get_retention_analysis
3. 当用户问到「高价值客户/大客户/VIP/客户分层/RFM」时，调用 get_rfm_analysis
4. 当用户需要「具体客户名单/找出哪些客户/客户列表」时，调用 get_customer_list
5. 当用户问「有没有问题/有什么异常/需要关注什么」时，调用 get_anomaly_detection
6. 当用户问到「业绩/订单量/销售额/趋势」时，调用 get_order_trends
7. 当用户问到「LTV/客户价值/高潜力客户/续费价值」时，调用 get_ltv_analysis
8. 当用户问到「活动/营销/大促/转化率/活动效果」时，调用 get_campaign_analysis
9. 如果一个问题同时涉及多个维度，可以并行调用 2-3 个工具，综合结果后统一回答
10. 如果用户的问题超出上述工具的覆盖范围，明确告知「当前版本暂不支持查询 [具体内容]」，不要编造数据

## 时间范围处理
- 用户说「最近」「近期」时，默认使用近 30 天
- 用户说「上周」时，使用上一个完整自然周（周一至周日）
- 用户说「本月」时，使用当月 1 日至今
- 如果用户的时间表述模糊，主动说明你使用的时间范围（例如「以下是近 30 天的数据：」）

## 输出格式规范
- 始终用中文回答（除非用户明确要求英文）
- 关键数字使用**加粗**标注
- 多个指标用分隔列表而非连续长句
- 每次回答末尾提供 1-2 个追问建议（「你还可以问：…」），引导用户深入探索
- 对于异常情况，在数字前加 ⚠️ 标注
- 回答长度适中：汇总摘要控制在 150 字以内，详细分析不超过 400 字
- 不要暴露工具调用的内部名称（如不要说「我调用了 get_rfm_analysis」）

## 安全与边界
- 只返回本企业的聚合统计数据，不展示具体消费者的姓名、手机号或身份证等个人信息
- 如果查询返回空数据，告知用户「暂无符合条件的数据」，并建议调整查询条件
- 禁止回答与客户数据分析无关的问题（如通用知识问答、其他系统操作等）
```

#### Instructions 设计决策说明

| 设计选择 | 理由 |
|---|---|
| 精确的工具路由规则（Rule 1-10）| M365 Copilot 的工具路由基于语义相似度，规则描述避免了歧义触发（如 overview 和 orders 都能回答「业绩」类问题） |
| 时间范围默认规则 | 用户最常见的错误是时间范围不明确，统一默认值减少工具调用参数错误 |
| 追问建议 | 引导多轮对话，提升用户使用深度，对应业务目标中「多轮对话率 ≥ 40%」 |
| 禁止暴露工具名称 | 降低技术感，保持对话自然；避免用户尝试「直接调用工具」绕过正常对话流程 |
| 长度限制 | M365 Copilot 在 Teams 界面的消息框有视觉限制，过长回答影响可读性 |

## 三、部署与接入体验

### 3.1 企业管理员安装配置流程（最少步骤设计）

**前置确认清单**（SocialHub 客成在对接初期完成）：
- SocialHub 销售确认该企业客户的 M365 Copilot 许可证覆盖范围
- SocialHub 运营在服务端配置 Entra Tenant ID → SocialHub Tenant ID 映射（2 分钟）
- 确认客户 IT 联系人拥有「Teams 管理员」或「全局管理员」角色

**IT 管理员操作（目标 4 步，30 分钟内完成）**：

```
步骤 1：下载 ZIP 包
访问 SocialHub 文档中心，下载最新版 socialhub-agent-vX.X.zip（通用包，所有客户共用）

步骤 2：上传到 Teams Admin Center
Teams Admin Center → Teams 应用 → 管理应用 → 上传自定义应用 → 选择 ZIP 文件

步骤 3：完成组织级 OAuth 预授权（Admin Consent）
在 Azure 门户 → 企业应用程序 → SocialHub Agent → 权限 → 授予管理员同意
（此步骤完成后，员工使用时无需单独授权）

步骤 4：配置部署范围
Teams Admin Center → 应用设置策略 → 选择目标用户组 → 添加 SocialHub Agent → 保存
（员工的 M365 Copilot 侧边栏将在 24 小时内自动显示 SocialHub Agent）
```

**关键设计决策**：步骤 3（Admin Consent 预授权）是减少员工首次使用摩擦的核心操作。在部署文档中将此步骤标记为「强烈推荐」，并提供 Azure 门户操作的截图指引。如果 IT 管理员跳过此步骤，员工首次点击 Agent 时会弹出授权窗口，对管理层用户可能造成流失。

### 3.2 企业用户首次使用体验

**场景：IT 已完成 Admin Consent 预授权**

```
1. 员工打开 Teams 或 M365 Copilot Chat
2. 左侧边栏出现「SocialHub 客户智能助手」图标（由 IT 推送）
3. 点击图标，进入专属 Copilot Chat 界面
4. 界面展示 6 个 Conversation Starters（预设问题卡片）
5. 用户点击任意 Starter 或直接输入问题
6. 5 秒内获得第一条数据回答
```

**场景：IT 未完成 Admin Consent（员工需自助授权）**

```
1-4. 同上
5. 点击 Starter 后，弹出「需要授权访问 SocialHub 数据」提示框
6. 用户点击「使用公司账号授权」→ 跳转到 Microsoft 登录页（已预填企业账号）
7. 用户点击「接受」→ 一次性授权完成
8. 自动返回对话界面，5 秒内获得数据回答
9. 此后使用无需再次授权（token 静默刷新）
```

**首次使用体验优化细节**：
- Conversation Starters 显示在对话框上方，不需要用户知道可以问什么
- Agent 头像使用 SocialHub 品牌图标（192×192 彩色图标），与企业内部其他 Agent 视觉区分
- 第一次回答末尾附加「你可以在任何时候用自然语言向我提问，或点击上方的快捷问题开始」的引导文字

### 3.3 认证策略产品决策

#### 结论：MVP 使用 API Key via ATK（非 auth:None），生产正式使用 Entra ID OAuth

| 方案 | 选择 | 理由 |
|---|---|---|
| `auth: None` + IP 白名单 | **仅限内测 Phase 0** | IP 白名单在云平台动态 IP 环境下不可靠；M365 Copilot 的出口 IP 段会随微软基础设施变化，维护成本高且安全性弱，不适合给真实企业客户使用 |
| API Key via ATK | **MVP Phase 1-2（内测+EA）** | 实现简单（1 天工程量），多租户支持（API Key = tenant 绑定），用户无感知（管理员配置一次），适合 5-10 个 EA 客户的快速验证，风险可控 |
| Entra ID OAuth | **生产标准（EA 结束前完成）** | 官方推荐，AppSource 上架要求，员工离职自动失权，审计日志完整，企业 IT 安全审查友好 |

#### API Key 方案的具体实现决策

- API Key 由 SocialHub 运营生成并通过安全渠道（加密邮件或 SocialHub 客户后台）发送给企业 IT 管理员
- API Key 在 M365 Agents Toolkit 的 `env/.env.development` 中注入，管理员配置一次，员工不可见
- 服务端维护 `API_KEY → tenant_id` 映射表，禁止客户端传入 tenant_id 参数
- API Key 轮换周期：每 6 个月或员工离职涉及 Key 泄露时主动轮换

#### 不选择 auth:None 的核心理由

`auth: None` 在 02-business-design.md 的「内测阶段」中被标注为可用，但产品层面不推荐作为 MVP 方案，原因是：
1. 真实 EA 客户的 IT 安全团队会问「访问是否有认证」——无认证答案会阻碍合同签署
2. M365 Copilot 的出口 IP 范围不公开文档化，IP 白名单维护是持续运营负担
3. API Key 方案的额外工程量不超过 1 天，没有理由接受更低安全性的方案

## 四、MVP 范围界定

### MVP 必须包含（不可裁剪）

| 类别 | 内容 | 理由 |
|---|---|---|
| **核心工具集** | 8 个精选 MCP 工具（P0 + P1 全部） | 覆盖 80% 的用户查询场景，缺少任何一个 P0 工具会导致关键场景失效 |
| **HTTP Transport** | MCP Server 支持 HTTP Streamable Transport | M365 Copilot 技术前提，无此能力则整个集成不可能实现 |
| **API Key 认证** | 服务端 API Key 验证 + tenant_id 注入 | 多租户安全隔离的最低要求，无认证不可给真实客户使用 |
| **Teams App 包** | manifest.json + declarativeAgent.json + plugin.json | 企业管理员部署的唯一入口 |
| **Conversation Starters** | 6 个预设问题 | 解决「用户不知道能问什么」的首次使用障碍，是留存率的关键因素 |
| **中文 Instructions** | Agent 指令（2.4 节完整版本） | 确保工具路由准确性和输出格式规范，是用户体验的核心 |
| **Render Starter 部署** | MCP Server 部署在 Render Starter 层（非 Free） | Render Free 层冷启动 60 秒，超过 M365 Copilot 超时阈值，必须使用 Starter |
| **自有域名** | `mcp.socialhub.ai`（或等效域名）的 HTTPS 端点 | 避免 Render 迁移时所有企业客户 ZIP 包失效（详见迭代记录 Round 3） |
| **Admin Consent 部署文档** | 含截图的 IT 部署指引 | 没有文档，企业 IT 不会主动找 Azure Admin Consent 选项，导致管理层用户首次体验差 |
| **健康检查端点** | `/health` HTTP 接口 | Render 部署和运营监控的基础设施必需 |

### 明确排除（后续迭代）

| 排除内容 | 排除理由 | 后续迭代时机 |
|---|---|---|
| Entra ID OAuth（OAuthPluginVault） | EA 阶段 API Key 已足够；OAuth 额外 2-3 天工程量，且需要 Azure AD 应用注册（有额外配置步骤）；GA 前必须完成 | EA 结束前，Phase 3 |
| `get_store_analysis` / `get_product_analysis` | 行业特化工具，MVP 阶段用户基数小，优先验证通用场景 | 连锁零售专项版本 |
| 主动推送/定时摘要 | 需要额外的 Scheduler 基础设施；Declarative Agent 本身不支持主动推送，需要 Bot 架构 | 功能扩展版本 |
| AppSource 上架 | 需要微软合规认证流程（数周），超出 MVP 时间窗口 | GA 阶段 |
| 企业内部岗位级权限分级 | 聚合统计数据合规风险低，额外权限系统显著增加部署复杂度（详见 02-business-design.md） | 根据客户明确需求决定 |
| 多语言支持（英文界面） | 首批目标客户为中文工作环境；中英文双语工具描述在内测数据验证后再添加 | EA 后期 |
| Copilot Studio 自定义 UI | 超出 Declarative Agent 范围，需要 Power Platform 授权 | 长期路线图 |
| 离线/导出功能 | 需要额外的数据导出 API，超出当前 MCP 工具设计 | 根据用户反馈决定 |

## 五、非功能需求

### 5.1 响应时间目标

| 场景 | 目标值 | 说明 |
|---|---|---|
| **单工具调用 P50** | < 3 秒 | 含 MCP Server 工具执行时间 + Copilot LLM 处理时间，用户感知「近实时」 |
| **单工具调用 P95** | < 5 秒 | M365 Copilot 工具调用超时阈值约为 30 秒，但用户体验要求更严格 |
| **双工具并行调用 P95** | < 8 秒 | 部分复杂问题触发 2 个工具，MCP Server 需支持并行处理 |
| **冷启动（Render Starter）** | < 5 秒 | Render Starter 层冷启动约 3-5 秒，符合要求（Free 层 60 秒不可接受） |
| **缓存命中响应 P95** | < 1 秒 | 15 分钟 TTL 缓存命中时，MCP Server 不访问上游，响应极快 |

**实现保障**：
- Render Starter 层（非 Free）消除冷启动问题
- 现有 15 分钟 TTL 缓存 + 并发去重机制零改动继续生效
- `stateless_http=True` 确保多实例扩容不引入状态同步延迟

### 5.2 可用性要求

| 指标 | 目标值 | 说明 |
|---|---|---|
| **月度可用性** | ≥ 99.5% | 允许每月约 3.6 小时的计划或非计划停机 |
| **单次故障最大时长** | < 30 分钟 | 超过 30 分钟需主动通知企业客户 IT 管理员 |
| **OAuth 授权成功率** | ≥ 95% | 首次登录失败率控制在 5% 以内 |
| **工具调用成功率** | ≥ 99% | 工具调用失败（非数据为空）触发告警 |

**降级预案**：
- MCP Server 不可用时，Agent 返回固定提示「SocialHub 数据服务暂时不可用，预计恢复时间 XX 分钟，请稍后重试」，不暴露技术错误信息
- Render 平台故障时，通过 DNS CNAME 切换到备用部署（要求自有域名，详见迭代记录）

**监控告警**：
- `/health` 接口每 5 分钟由 Render 健康检查探测
- 工具调用失败率超过 1% 时触发告警
- P95 响应时间超过 5 秒时触发性能预警

### 5.3 数据安全约束

| 约束 | 实现方式 | 不可妥协 |
|---|---|---|
| **租户数据强隔离** | `tenant_id` 从服务器验证的凭证（API Key 或 OAuth token）中提取，客户端传入的 tenant_id 参数一律忽略 | 是 |
| **缓存 Key 含租户隔离** | `_cache_key()` 函数必须将 `tenant_id` 纳入 Key 计算，防止不同租户的缓存数据互污 | 是 |
| **传输层加密** | 全程 HTTPS（Render 自动提供 TLS），MCP Server 拒绝 HTTP 明文请求 | 是 |
| **不存储查询结果** | MCP Server 只做计算和转发，查询结果不持久化到服务器文件系统或数据库 | 是 |
| **最小 OAuth Scope** | Entra ID OAuth 只申请 `openid profile`，不申请任何 Graph API 权限；不读取企业 SharePoint/邮件 | 是 |
| **PII 不出现在回答中** | 工具调用结果中如含有消费者手机号、身份证等字段，MCP Server 在返回前脱敏（仅保留聚合统计） | 是 |
| **API Key 安全传输** | API Key 不通过邮件明文传输，通过 SocialHub 客户后台安全链接或加密邮件发送 | 是 |
| **员工离职自动失权** | OAuth 方案下，Entra 账号禁用后 token 自动失效；API Key 方案下，企业 IT 需要通知 SocialHub 吊销 Key | OAuth 是；API Key 需运营流程保障 |

## 六、迭代记录

### Round 1：挑剔的企业运营总监视角

**视角设定**：我是某零售集团的运营总监，手下有 30 人的运营团队，每天在 Teams 里处理大量消息，刚听说公司要部署这个 SocialHub Agent。

**发现的问题**：

**问题 1：Conversation Starters 不够「运营」**

初版的 6 个 Starter 太「产品化」，像功能介绍而非真实问题。「帮我查一下 RFM 分析」这类措辞没有运营同学会用——她们会说「哪些客户快流失了」或「高价值用户最近怎么了」。

**修正**：重写全部 6 个 Starter，使用运营人员的实际口语，避免出现「RFM」「LTV」等技术术语（这些词可以出现在回答里，但不应出现在问题里）。

**问题 2：能力边界里缺少「不能做什么」的说明**

用户会问「能帮我给这些客户发优惠券吗」「能把这个结果导出成 Excel 吗」——如果 Agent 只是沉默或报错，用户体验极差。必须设计明确的「边界告知」机制。

**修正**：在 Instructions 中增加「超出范围时的固定回答模板」，在能力边界表中明确列出「不能做什么」及替代路径。

**问题 3：多轮对话体验未设计**

初版方案只描述了单轮查询，但运营团队实际使用高度依赖追问：「那这些客户里有多少是上个月新增的？」「再细看一下北区的数据」。没有多轮追问的设计，Agent 就是一个高级搜索框，而不是分析助手。

**修正**：在 Instructions 中明确「追问建议」机制（每次回答末尾提供 1-2 个相关追问）；在用户故事中将「多轮追问」作为 US-02 的验收标准之一。

### Round 2：业务方视角对照 02-business-design.md

**视角设定**：我是 SocialHub 的产品 BD，手里拿着 02-business-design.md 的业务设计文档，逐项核对产品方案是否忠实反映业务意图。

**发现的问题**：

**问题 1：认证策略产品决策与业务设计有偏差**

业务设计文档（02-business-design.md 第三节）明确指出「企业内不做岗位级权限分级」，并强调「Entra ID token 中提取 tenant_id」是生产方案。但初版产品设计中，MVP 阶段使用 API Key，导致「员工离职失权」等安全约束只能通过运营流程（而非技术手段）保障。这与业务设计的安全承诺不完全一致。

**修正**：在认证策略决策中明确说明「API Key 仅为 EA 阶段临时方案，EA 结束前必须完成 Entra ID OAuth 升级」，并将 OAuth 升级明确标注为「GA 必须条件」而非「后续迭代可选项」。

**问题 2：工具描述质量的重要性在产品方案中被低估**

02-business-design.md Round 1 修正中明确指出「工具描述质量是真正的瓶颈，不仅仅是工具数量」。但初版产品方案中，工具描述迭代没有出现在任何验收标准或运营计划中。

**修正**：在 MVP 范围中增加「工具描述迭代」作为内测阶段的必做工作项：收集用户实际提问语料，对照工具路由结果做 description 优化，这与选哪 8 个工具同等重要。在 Instructions 中增加工具路由的精确触发规则（2.4 节 Rule 1-10），减少对工具 description 语义匹配的依赖。

**问题 3：目标市场限定未反映在产品设计中**

业务设计文档明确指出（Round 1 修正）：目标客户限定为「已购 M365 Copilot 的企业」，主要面向出海企业或跨国企业的亚太区团队。但产品方案中没有体现这个市场定位限定，容易导致销售团队向不用 M365 的客户推广。

**修正**：在部署文档的前置确认清单中增加「确认企业客户已购买 M365 Copilot 许可证，且许可证覆盖目标用户群」，并在业务约束中明确这是准入条件。

**问题 4：度量体系与业务设计 KPI 对齐**

业务设计文档第七节有详细的三层度量体系（技术健康、使用深度、业务结果），但产品方案中缺少对应的度量指标。

**修正**：在非功能需求章节的可用性指标中，对齐业务设计中的具体目标值（如 P95 < 5 秒、OAuth 授权成功率 ≥ 95%、月活率 ≥ 70%）。

### Round 3：工程师视角

**视角设定**：我是负责实现这个功能的后端工程师，需要评估产品方案的可实现性和工程合理性。

**发现的问题**：

**问题 1：「并行工具调用」的产品需求需要工程确认**

Instructions 设计中写道「可以并行调用 2-3 个工具」，但 M365 Copilot 的工具调用是否支持并行（同时发出多个 MCP 请求）取决于 Copilot 的运行时实现，而非 MCP Server 端的能力。如果 Copilot 运行时是串行调用，「并行 2-3 个工具」就是错误的产品承诺。

**修正**：将「并行调用」改为「顺序调用多个工具」，并调整 US-05 的验收标准中的多轮总耗时从「15 秒」调整为「针对串行调用实际测量后确定」。Instructions 中保留「可以调用 2-3 个工具」但去掉「并行」的表述，实际行为由 Copilot 运行时决定。

**问题 2：`_cache_key()` 加入 tenant_id 是安全必改项，但未出现在产品需求中**

这个改动在 summary.md 中被标注为「安全必改」，但在产品方案的 MVP 必须包含清单中没有对应条目，可能在工程实现中被遗漏。

**修正**：在 MVP 必须包含清单中显式加入「缓存 Key 含租户隔离」作为数据安全约束，并在非功能需求的数据安全约束表中标注「不可妥协」。

**问题 3：「14 个工具中排除 8 个」的逻辑不够清晰**

产品方案选了 8 个工具，但工程师实现时需要知道：其余 28 个工具是完全不部署（不放入 plugin.json），还是部署了但不在 Instructions 中描述？两种方案对 token 预算和工具路由准确性影响不同。

**修正**：明确产品决策——**只将 8 个核心工具的描述放入 mcp-tools.json**（通过 ATK 生成时手动筛选），MCP Server 的完整 36+ 工具仍然存在（不删除代码），但 M365 Copilot 运行时只能发现 8 个工具。这样既保留了未来扩展能力，又控制了 token 预算。

**问题 4：非功能需求中的 P95 < 5 秒目标是否包含 LLM 处理时间？**

MCP Server 自身的 P95 响应时间（工具执行）和 M365 Copilot 的端到端响应时间（含 LLM 推理）是两个不同的指标。产品方案混淆了这两个概念，给工程师造成误导。

**修正**：区分两个响应时间目标：（1）MCP Server P95 工具执行时间 < 2 秒（SocialHub 可控）；（2）用户感知的端到端响应时间（含 LLM 推理）P95 < 10 秒（M365 Copilot 侧，SocialHub 不可控，仅作参考目标）。5 秒目标改为 MCP Server 自身指标。

**问题 5：自有域名（mcp.socialhub.ai）是「产品决策」还是「运营建议」？**

在 02-business-design.md Round 3 中，自有域名被定义为避免 ZIP 包大规模失效的「基础设施锚点」。但在当前产品方案中，这个决策只出现在 MVP 清单中，没有足够的理由说明，工程师可能跳过。

**修正**：在 MVP 必须包含清单中将「自有域名」标注为「不可裁剪」，并明确说明理由：ZIP 包中硬编码的 MCP Server URL 一旦变更，所有已部署的企业客户需要重新上传 ZIP 包，这是一个随客户数量线性扩大的运营风险。自有域名通过 DNS CNAME 解耦 URL 与实际部署，是一次性基础设施投入（$12/年），避免未来大规模运营成本。
