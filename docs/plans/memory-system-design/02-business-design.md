## 业务目标与优先级

### 根本业务问题

SocialHub CLI 的 AI 助手存在"永久失忆"问题：每次对话从零开始，无法积累对用户和企业业务的认知。这导致：

1. 用户每次必须重复说明自己的分析偏好（"按渠道拆分"、"看最近 7 天"）
2. AI 不知道企业历史营销活动，无法将指标波动与活动背景关联
3. 过去分析产生的有价值洞察（"女装周末转化率高 40%"）只活在终端，当次关闭即消失
4. AI 不知道已安装哪些 Skills，无法主动推荐最合适的工具

### 业务目标层级

**P0 — 核心痛点（必须解决，系统价值基础）**

| 目标 | 业务价值 | 成功指标 |
|------|---------|---------|
| T1：偏好记忆 | 消除每次重复说明的摩擦，降低操作负担 | 重复性偏好说明减少 ≥80% |
| T2：洞察持久化 | 让 AI 产出的分析结论不再秒消失，形成知识积累 | 会话洞察保留率 100% |
| T3：业务上下文注入 | AI 了解企业营销活动背景，避免将活动驱动的指标波动误判为异常 | 活动期间误判率降低 ≥60% |

**P1 — 重要增强（显著提升 AI 对话质量）**

| 目标 | 业务价值 | 成功指标 |
|------|---------|---------|
| T4：会话摘要接续 | 支持跨会话讨论上次分析的结论，而不是从头重问 | 跨会话上下文接续率 ≥70% |
| T5：Skills 感知 | AI 知道可用 Skills 并主动推荐，减少用户"我能用什么工具"的困惑 | Skills 主动推荐准确率 ≥75% |

**P2 — 长期价值（形成竞争护城河）**

| 目标 | 业务价值 | 成功指标 |
|------|---------|---------|
| T6：团队知识库 | 个人洞察 → 团队共享，形成组织记忆 | （后续版本，本期设计预留接口） |
| T7：异常基线 | AI 基于历史均值判断波动是否异常，而不是给通用回答 | （后续版本） |

## 核心业务逻辑（写入 / 读取 / 触发机制）

### 业务逻辑全景图

```
用户输入
    │
    ▼
[启动阶段] MemoryManager.load()
    ├── 读取 user_profile.yaml（偏好层）
    ├── 读取 business_context.yaml（业务上下文层）
    ├── 读取最近 N 条 session_summaries（情节记忆层）
    └── 读取最近 M 条 analysis_insights（洞察层）
    │
    ▼
[注入阶段] build_system_prompt(memory_context)
    └── 动态构建 SYSTEM_PROMPT，注入记忆上下文（token 预算控制）
    │
    ▼
[对话执行] AI 对话（现有 executor/insights 链路）
    │
    ▼
[写入阶段 — 会话结束时触发]
    ├── 自动：insights.py 生成的洞察 → write_insight()
    ├── 自动：会话消息摘要提炼 → write_session_summary()
    ├── 自动（建议确认）：AI 检测到偏好变化 → suggest_preference_update()
    └── 手动：用户执行 sh memory set / sh memory init
```

### 写入触发规则

#### 自动写入（无需用户确认）

| 触发时机 | 写入内容 | 触发位置 |
|---------|---------|---------|
| 每次 smart-mode 会话正常结束 | 会话摘要（执行了什么命令、得到什么结论） | `main.py` 会话退出钩子 |
| `insights.py` 生成洞察后 | 洞察内容（量化结论、异常发现） | `insights.py` 末尾钩子 |
| 活动期间执行相关分析后 | 与活动关联的效果数据（AI 自动提炼聚合值） | `insights.py` 特殊路径 |

**自动写入约束**：仅写入聚合/统计性结论，禁止写入任何个人级数据。写入前必须过 PII 扫描。

#### 需用户确认写入（AI 建议 + 用户确认）

| 触发时机 | AI 行为 | 用户操作 |
|---------|---------|---------|
| AI 观察到用户连续 3 次以上使用相同维度（如每次都加 `--channel`） | 终端提示："我注意到您经常按渠道分析，是否要将渠道设为默认分析维度？[y/N]" | y → 写入偏好；N → 本次忽略，30 天内不再提示 |
| 执行完成后 AI 洞察包含有价值的基准值（如"本季度日均 GMV 约 50 万"） | 终端提示："是否将此基准值记入业务上下文，用于后续异常判断？[y/N]" | y → 写入 business_context；N → 仅保留在当次洞察中 |

**不允许 AI 静默修改偏好层和业务上下文层**——这两层是用户信任基础，必须透明。

#### 完全手动写入

| 命令 | 说明 |
|-----|------|
| `sh memory init` | 首次使用引导，交互式问卷写入 user_profile |
| `sh memory set <key> <value>` | 直接更新偏好字段 |
| `sh memory add-context <key> <value>` | 手动添加业务上下文条目 |
| `sh memory add-campaign` | 交互式添加营销活动记录 |

### 读取触发规则

| 读取层 | 触发时机 | 读取条件 |
|-------|---------|---------|
| L4 偏好层（user_profile） | 每次 AI 对话前 | 文件存在时必加载，注入 SYSTEM_PROMPT 顶部 |
| L4 业务上下文层（business_context） | 每次 AI 对话前 | 文件存在时必加载 |
| L3 洞察层（analysis_insights） | 每次 AI 对话前 | 最近 30 天内的洞察，最多 5 条，按时间倒序 |
| L2 会话摘要层（session_summaries） | 每次 AI 对话前 | 最近 7 天内的摘要，最多 3 条 |
| L1 工作记忆（session） | 对话中 | 已有机制（不变） |

**读取不命中（冷启动）**：文件不存在或为空时，AI 以无个性化模式运行，引导用户执行 `sh memory init`。

## 业务数据模型

### 总览

```
~/.socialhub/memory/
├── user_profile.yaml          # L4 偏好层（个人，永久）
├── business_context.yaml      # L4 业务上下文层（企业，可 git 共享）
├── analysis_insights/         # L3 语义记忆（90 天有效）
│   └── {date}-{slug}.json
└── session_summaries/         # L2 情节记忆（30 天有效）
    └── {session_id}.json
```

---

### 模型 1：用户偏好（user_profile.yaml）

**业务定义**：描述该用户的分析习惯和输出偏好。这是"我这个人通常怎么工作"的静态画像，跨业务周期长期有效，是 AI 个性化体验的基础。

**写入主体**：用户手动（`sh memory init` / `sh memory set`）或 AI 建议后用户确认。  
**更新频率**：低频（偏好形成后相对稳定，月级别可能更新 1-2 次）。

```yaml
# ~/.socialhub/memory/user_profile.yaml
version: "1"
created_at: "2026-04-02T10:00:00Z"
updated_at: "2026-04-02T10:00:00Z"

role: "运营经理"                    # 用户职责角色

analysis:
  default_period: "7d"             # 默认时间窗口（7d / 30d / 90d）
  preferred_dimensions:            # 常用分析维度，注入 AI 时优先建议
    - channel
    - province
  rfm_focus:                       # 重点关注的 RFM 分层
    - At-Risk
    - Champions
  scope:                           # 责任域过滤（只看这些渠道/分类）
    channels:
      - 天猫
      - 京东

output:
  format: "table"                  # table / json / csv
  precision: 1                     # 小数位数
  include_yoy: true                # 是否默认附带同比

skills:
  frequently_used:                  # AI 自动统计，读多少次累积（只记名称）
    - rfm_analysis
    - channel_breakdown
```

**业务约束**：
- `scope.channels` 只记渠道名称（天猫、京东），不记任何与客户个人相关的数据
- `skills.frequently_used` 只记 Skill 名称，不记使用时的输入参数（参数可能含 PII）

### 模型 2：业务上下文（business_context.yaml）

**业务定义**：描述这家企业的业务背景和运营节奏。这是"我们公司的生意是怎么运作的"的企业级知识，供 AI 理解分析背景、避免脱离业务实际给出通用建议。团队可通过 git 共享。

**写入主体**：用户手动（`sh memory add-context` / `sh memory add-campaign`）或 AI 建议后用户确认。  
**更新频率**：中低频（随业务周期演进，季度级别可能更新）。

```yaml
# ~/.socialhub/memory/business_context.yaml
version: "1"
created_at: "2026-04-02T10:00:00Z"
updated_at: "2026-04-02T10:00:00Z"

business:
  primary_category: "服装"           # 主营品类
  key_metrics:                       # 企业核心 KPI（影响 AI 默认关注哪些指标）
    - GMV
    - 转化率
    - 客单价
    - 复购率
  fiscal_year_start_month: 1         # 财年起始月

campaigns:                           # 历史营销活动（时效性强，AI 关联分析时使用）
  - id: "ACT001"
    name: "2026 三八大促"
    period: "2026-03-08/2026-03-15"
    channels:
      - 天猫
      - 京东
    effect_summary: "GMV 环比+15%，转化率+8%"    # 聚合效果，无个人数据
    notes: "女装品类表现最强"

baselines:                           # 业务基准值（用于 AI 判断波动是否异常）
  gmv_daily_avg_30d: 500000          # 近 30 日日均 GMV（用户确认后写入）
  updated_at: "2026-04-01"
```

**业务约束**：
- `campaigns` 只记录聚合效果（GMV 变化率、转化率），不记录具体订单数据或参与客户信息
- `baselines` 的数值来自用户确认，不允许 AI 静默写入（避免 AI 误判导致基线失真）

### 模型 3：分析洞察（analysis_insights/{date}-{slug}.json）

**业务定义**：某次具体分析会话中，AI 产出的有价值业务发现。是"某天发现了什么"的时效性知识，随时间衰减但可积累形成知识库。主要服务于接续性分析（"上次分析发现 X，这次继续深入"）。

**写入主体**：AI 自动（`insights.py` 执行结束后触发），无需用户确认。  
**TTL**：90 天活跃期，之后降权保留（不再注入 SYSTEM_PROMPT，但不删除），180 天后清理。

```json
{
  "id": "2026-04-02-channel-gmv-analysis",
  "session_id": "2026-04-02T09-30-00-abc123",
  "created_at": "2026-04-02T10:15:00Z",
  "ttl_days": 90,

  "query_intent": "渠道 GMV 分析",
  "time_window": "2026-03-01/2026-03-31",
  "dimensions_used": ["channel"],

  "key_findings": [
    "天猫渠道 3 月 GMV 占比达 62%，环比上升 8 个百分点",
    "京东渠道客单价高于天猫 22%，但订单量少 40%"
  ],
  "anomalies": [
    "3 月 15 日前后天猫 GMV 异常高峰，关联三八大促（ACT001）"
  ],

  "commands_executed": [
    "sh analytics overview --period=30d --dimension=channel"
  ],

  "confidence": "high",
  "pii_scanned": true,
  "content_hash": "sha256:abc123..."
}
```

**业务约束**：
- `key_findings` 和 `anomalies` 只允许包含聚合统计结论，写入前必须过 PII 扫描
- `commands_executed` 只记录命令名和标志，不记录参数值（参数可能含业务敏感信息如具体客户 ID）
- `content_hash` 用于审计，AI 不使用此字段

---

### 模型 4：会话摘要（session_summaries/{session_id}.json）

**业务定义**：某次完整对话会话的浓缩摘要。服务于"上次我们聊到哪了"的跨会话接续需求。存储的是对话结构化摘要，而非原始消息（原始消息在 session 层，24h 后自然过期）。

**写入主体**：AI 自动（会话正常结束时触发，LLM 批量提炼），无需用户确认。  
**TTL**：30 天。

```json
{
  "session_id": "2026-04-02T09-30-00-abc123",
  "created_at": "2026-04-02T10:20:00Z",
  "ttl_days": 30,

  "topic": "Q1 渠道销售复盘",
  "duration_minutes": 22,
  "turn_count": 8,

  "commands_run": 3,
  "skills_used": ["channel_breakdown"],

  "summary": "用户分析了 Q1 各渠道 GMV 表现，重点关注天猫和京东对比。发现三八大促对天猫 GMV 有显著提升。讨论了是否需要调整 Q2 渠道投入策略。",

  "open_questions": [
    "Q2 是否需要提升京东渠道投入？（用户尚未决策）"
  ],

  "pii_scanned": true,
  "content_hash": "sha256:def456..."
}
```

**业务约束**：
- `summary` 由 LLM 提炼，必须要求 LLM 遵守"只描述分析行为和聚合结论，不包含具体客户信息"
- `open_questions` 只记录业务决策问题，不记录任何涉及具体人员的问题

## 业务约束

### PII 边界

#### 允许写入记忆的数据类型

| 类型 | 示例 | 允许原因 |
|------|------|---------|
| 用户操作偏好 | `default_period: "7d"` | 操作习惯，无客户个人信息 |
| 渠道/维度名称 | `channels: ["天猫", "京东"]` | 平台名称，非个人信息 |
| 聚合统计结论 | `"渠道A GMV 占比 62%"` | 汇总值，无法还原个人数据 |
| 活动聚合效果 | `"GMV 环比 +15%"` | 汇总效果，无交易明细 |
| Skill 名称 | `"rfm_analysis"` | 工具元数据，无业务数据 |
| 业务品类标签 | `primary_category: "服装"` | 企业属性，非个人信息 |

#### 绝对禁止写入的数据类型

| 类型 | 示例 | 禁止原因 |
|------|------|---------|
| 客户手机号 | `13812345678` | PIPL 个人信息 |
| 客户身份证号 | `110101199001011234` | PIPL 敏感个人信息 |
| 客户邮箱 | `user@example.com` | PIPL 个人信息 |
| 客户姓名 | `张三` | PIPL 个人信息 |
| 订单号（16 位+）| `1234567890123456` | 可关联到具体交易和个人 |
| 任何 SELECT 查询原始结果 | `[{name: "张三", phone: "..."}]` | 含个人级明细数据 |
| 客户 ID 集合 | `customer_ids: [1001, 1002]` | 可间接关联到个人 |

**执行机制**：记忆写入前必须调用 `_mask_pii()` 扫描。扫描到 PII 标识（`[PHONE_MASKED]` 等）时，该条记忆**整体丢弃**（不是脱敏后写入），并记录审计事件。

---

### 信任边界：AI 何时可自动更新，何时需用户确认

| 记忆层 | AI 自动写入 | 需用户确认 | 禁止 AI 写入 |
|-------|-----------|----------|------------|
| L4 用户偏好（user_profile） | 无 | AI 观察到连续偏好信号后建议，用户确认 | 直接修改 |
| L4 业务上下文（business_context）| 无 | AI 提炼聚合基准值后建议，用户确认 | 直接修改 |
| L3 分析洞察（analysis_insights） | 会话中由 insights.py 触发，自动写入 | 无需确认（纯 AI 产出，非用户偏好） | — |
| L2 会话摘要（session_summaries） | 会话结束时自动提炼写入 | 无需确认 | — |

**设计哲学**：AI 自动写入自己的"分析笔记"（L2/L3），但不得擅自修改用户的"操作习惯设定"（L4 偏好层）。用户是自己偏好的唯一权威。

---

### 时效约束

| 记忆层 | 有效期 | 业务逻辑依据 |
|-------|-------|------------|
| L4 用户偏好 | 永久（用户主动更新） | 分析习惯长期稳定，无需自动失效 |
| L4 业务上下文 | 永久，但 campaigns 按日期自然降权 | 企业背景稳定；活动过期后降权（不注入，但不删除）|
| L3 分析洞察 | 活跃期 90 天（权重全），90-180 天降权保留 | 电商季度业务周期约 90 天，超出后洞察时效性下降 |
| L2 会话摘要 | 30 天 | 短期工作记忆，超出后接续价值低 |

**降权机制（区别于删除）**：过期洞察不再注入 SYSTEM_PROMPT，但保留在本地文件。用户通过 `sh memory show --all` 仍可查看历史洞察，AI 不会主动使用。

**活动自动降权规则**：`business_context.campaigns` 中的活动，当 `period.end` 距今超过 90 天时，在注入 SYSTEM_PROMPT 时自动跳过（不注入）。用户可手动保留重要历史活动作为对比基准。

## 度量体系

### 核心业务价值度量

#### 使用效率类（系统核心目标 T1）

| 指标 | 计算方式 | 目标值 | 采集位置 |
|-----|---------|-------|---------|
| 偏好重复说明率 | 每次对话中用户输入含"按渠道/按品类/7天/..."等偏好关键词，且 user_profile 已有对应记录的频率 | 降低 ≥80% | ai_trace.jsonl 关键词统计 |
| 命令参数补全率 | AI 基于记忆自动填充参数（而非用户显式指定）的比例 | ≥60% | executor.py 参数来源标记 |

#### AI 质量类（系统核心目标 T2/T3）

| 指标 | 计算方式 | 目标值 | 采集位置 |
|-----|---------|-------|---------|
| 洞察持久化率 | insights.py 产出次数 vs 成功写入 analysis_insights 次数 | 100% | MemoryStore 写入日志 |
| 活动关联准确率 | 活动期间分析时，AI 主动关联活动背景的比例（用户评价） | ≥75% | 用户反馈命令 `sh memory feedback` |
| 跨会话接续成功率 | 用户明确引用上次分析，AI 正确接续的比例 | ≥70% | 用户反馈 |

#### 记忆健康度类（系统可持续性）

| 指标 | 计算方式 | 目标值 | 采集位置 |
|-----|---------|-------|---------|
| 记忆注入 Token 消耗 | 每次对话中记忆上下文占用的 Token 数 | ≤4000 token（总预算 12K 内） | ai_trace.jsonl |
| PII 扫描拦截率 | 被 PII 扫描拦截的记忆写入尝试次数 / 总写入尝试次数 | 目标 0%（触发即为设计缺陷信号） | 审计日志 |
| 记忆文件大小 | user_profile + business_context + 30 天内 insights/summaries 总大小 | ≤5MB | MemoryStore 定期检查 |

#### 用户信任类（核心体验保障）

| 指标 | 计算方式 | 目标值 | 采集位置 |
|-----|---------|-------|---------|
| 用户主动确认偏好建议率 | AI 建议更新偏好时，用户选择"y"的比例 | ≥60%（低于此值说明 AI 建议质量差）| sh memory 命令日志 |
| 用户手动纠正记忆频率 | 用户执行 `sh memory set` 修正 AI 写入内容的频率 | ≤1次/周（高于此值说明 AI 写入质量差）| 命令执行审计 |
| 记忆系统 NPS | 季度问卷："记忆系统让 AI 回答更准确了吗？（1-10）" | ≥8 | 用户主动反馈 |

### 反向指标（警戒线）

以下指标触发时需立即审查记忆系统：

| 警戒指标 | 阈值 | 含义 |
|---------|-----|------|
| AI 引用过期洞察率 | >5% | 洞察 TTL 降权机制失效 |
| 记忆写入失败率 | >1% | 基础设施问题（磁盘/权限） |
| Token 超预算率 | >5% 的对话超过 4000 token 记忆注入 | 注入策略需调整 |
| 用户主动关闭记忆率 | >10% | 用户不信任记忆系统（最严重信号） |

## 迭代记录（R1/R2/R3 对抗摘要及修正）

---

### R1：Devil's Advocate（可行性 / 逻辑矛盾 / 边界 / 过度设计）

**对抗质疑**

**质疑 1：过时记忆导致错误建议**

> "3 个月前的业务洞察显示'天猫占比 62%'，但业务格局已经改变（例如拼多多快速增长）。AI 如果继续引用旧洞察给出'重点投入天猫'的建议，将导致决策失误。"

初始设计问题：洞察层的 90 天 TTL 设计没有考虑"降权"与"全权重"的区分机制，只说了"90 天内有效"，但没有明确超过有效期后的处理方式，也没有设计洞察"新鲜度"提示。

**修正 R1-1**：
- 引入洞察降权机制（30-90 天权重衰减，90 天后不再注入 SYSTEM_PROMPT，但不删除）
- AI 注入洞察时，必须携带时间标签（"以下洞察来自 45 天前"），让用户能判断时效性
- `business_context.baselines` 写入时记录 `updated_at`，注入时 AI 必须告知基线更新时间

**质疑 2：AI 自动提炼偏好导致误判，用户不信任记忆系统**

> "用户某次分析因为数据问题临时用了 30 天窗口，AI 误判为'偏好 30 天'并建议更新偏好。用户一旦信任后点了 y，后续所有分析都错误地使用 30 天，直到用户手动发现并纠正。"

初始设计问题：偏好建议的触发条件描述为"连续 3 次"，但没有设计"连续"的定义（3 次在同一会话内？还是跨 3 个会话？），也没有排除"临时/调试性用法"的逻辑。

**修正 R1-2**：
- 偏好建议触发条件改为：**跨 3 个以上不同会话**，且每次都是同一维度（单会话内重复不算）
- 建议提示文案明确说明依据："我注意到您在过去 5 次不同的分析中都使用了 7 天时间窗口（最近一次：昨天），是否要将此设为默认？"
- 用户选择 N 后，**30 天内不再为该偏好提示**，避免骚扰

**质疑 3：10 种记忆类型是否全部值得实现，是否过度设计**

> "从用户看到 10 种记忆类型（偏好、上下文、洞察、摘要、Skills 使用频率、RFM 焦点、责任域、输出格式、活动背景、工作流别名），学习成本极高，维护成本也极高。MVP 阶段这样做是否合理？"

初始设计问题：虽然按 P0/P1/P2 分了优先级，但数据模型里把所有字段都展开了，没有明确"MVP 最小集"。

**修正 R1-3**：
- 明确 MVP 最小集（仅 3 个核心字段组）：`analysis.default_period`、`analysis.preferred_dimensions`、`output.format`
- user_profile.yaml 设计为可选字段，缺失时 AI 不报错，平稳降级为通用模式
- `skills.frequently_used`、`analysis.rfm_focus`、`business.baselines` 等进阶字段延迟到 v1.1 实现
- 工作流别名（第 10 种）移出记忆系统，通过现有 `heartbeat` 调度器实现

---

### R2："客户怎么用"

**场景 1：电商运营经理 — 第一天 vs 第 30 天**

**第一天体验（冷启动）**：
- 用户首次运行 `sh analytics overview`，AI 给出通用回答
- 终端底部提示："当前使用通用模式。运行 `sh memory init` (约 2 分钟) 即可开启个性化 AI 体验"
- 用户运行 `sh memory init`，回答 4 个问题（角色/时间窗口/常用维度/渠道范围）
- 下次运行时，AI 自动按渠道拆分、默认 7 天，用户无需再说

问题发现：如果 `sh memory init` 的问卷设计不友好，用户可能跳过，永远处于冷启动状态。

**修正 R2-1**：
- `sh memory init` 采用渐进式引导（第 1 个问题必填，后续全部可跳过），即使只回答 1 个问题也写入有效记忆
- 每次运行 `sh analytics` 类命令时，如果 user_profile 不存在，在命令结果底部显示一次提示（用弱提示而非阻断式）

**第 30 天体验**：
- AI 在分析三八大促后续数据时，自动说明："以下分析关联了您记录的三八大促活动（2026-03-08 至 03-15），大促期间 GMV 异常高峰已排除"
- 用户上周问了"Q1 渠道复盘"，这周问"Q2 应该怎么调整"，AI 说"根据上周的渠道复盘（天猫占比 62%，京东客单价更高）..."

问题发现：用户如何知道 AI 在用哪条记忆？如果记忆错了，用户无法定位。

**修正 R2-2**：
- AI 使用记忆时，在回答中以轻量标注形式说明来源（终端输出末尾附 `[memory: user_profile/渠道偏好]`）
- 增加 `sh memory show --used-in-last` 命令，显示"上次对话用了哪些记忆"

**场景 2：分析师 — 如何知道 AI 用了哪条记忆？记忆错了怎么纠正？**

问题发现：分析师需要更强的记忆透明度和纠错能力，但现有设计缺乏"记忆调试模式"。

**修正 R2-3**：
- 增加 `sh memory list --verbose` 命令，列出所有记忆条目及其最近一次被注入的时间
- 增加 `sh memory delete <id>` 命令，删除指定洞察或摘要条目
- AI 引用记忆时，在 `ai_trace.jsonl` 记录 `memory_referenced: ["insight/2026-03-20-channel"]`，分析师可通过 `sh history show <trace_id>` 查看

**场景 3：营销专员 — 让 AI 记住本次大促背景**

问题发现：活动背景写入流程不清晰。`sh memory add-campaign` 需要什么信息？格式是什么？

**修正 R2-4**：
- `sh memory add-campaign` 使用交互式问卷（活动名称 / 时间段 / 涉及渠道 / 预期目标），问卷完成后展示预览，用户确认后写入
- 写入后 AI 立即确认："已记录三八大促（2026-03-08 至 03-15），后续所有分析中遇到该时段的异常波动，我会自动关联此活动"
- 增加 `sh memory show campaigns` 命令，列出所有已记录活动

### R3："6 个月后维护者"

**质疑 1：记忆文件如何防止无限增长？**

> "分析师每天执行 3-5 次分析，每次产出 2 条洞察。6 个月后 `analysis_insights/` 目录将有约 360-600 个文件，每次启动 MemoryManager 都要扫描所有文件，性能会下降。"

初始设计问题：TTL 只说了"90 天降权，180 天清理"，但清理触发机制不明确（谁触发？什么时候执行？是否会影响启动性能？）。

**修正 R3-1**：
- 清理策略改为"惰性清理 + 写入前清理"（与现有 SessionStore.purge_expired() 一致），不在读取时扫描全目录
- 在每次 `MemoryStore.write()` 前（而非每次启动），检查是否有超过 180 天的文件需要清理
- 同时引入**文件数量上限**：`analysis_insights/` 最多保留 200 个文件（超出时删除最旧的），防止极端场景下无限增长
- L2 会话摘要同样限制：最多保留 60 个文件（约 2 个月的日常使用量）

**质疑 2：记忆质量如何保证？垃圾记忆和过时记忆的清理？**

> "AI 在某次有 bug 的分析后，自动写入了错误洞察（例如 'GMV 下降 50%'，实际是数据查询参数错误导致）。这条错误记忆会在接下来 90 天内持续影响 AI 的分析判断，用户几乎无法发现。"

初始设计问题：AI 自动写入的洞察没有"质量门槛"，也没有"用户可审阅"机制。

**修正 R3-2**：
- 引入洞察质量评分机制：`confidence` 字段（`high` / `medium` / `low`）
  - 执行成功且命令无报错 → `high`（自动注入 SYSTEM_PROMPT）
  - 执行有警告或数据量过小 → `medium`（注入时附加 "(注意: 此洞察基于有限数据样本)"）
  - 执行失败或结果为空 → `low`（不注入，仅本地保留供审查）
- 新增 `sh memory review` 命令：列出最近写入的洞察（特别是 medium/low 质量），用户可快速确认或删除
- AI 每次生成洞察时，在写入前验证：洞察文字中不包含"无数据"、"0 条记录"、"错误"等信号词（若包含则降级为 low）

**质疑 3：新功能（新命令、新分析维度）如何让记忆系统同步适配？**

> "6 个月后新增了 `sh analytics social-commerce` 命令（直播带货分析），但 user_profile.yaml 的 `preferred_dimensions` 没有对应字段，business_context.yaml 的渠道列表也没有'抖音'。记忆系统会静默失效，没有任何提示。"

初始设计问题：记忆模型是固定 schema，没有扩展机制。

**修正 R3-3**：
- `user_profile.yaml` 和 `business_context.yaml` 引入 `version` 字段，MemoryManager 启动时检查版本兼容性
- 记忆模型设计采用"宽松 schema"：已定义字段按规范处理，未知字段透传给 AI（不报错）
- 新命令上线时，对应的记忆字段在代码中用 `@since: v1.x` 标注，不强制要求旧用户迁移
- 提供 `sh memory migrate` 命令，检测当前记忆文件与最新 schema 的差异并提示用户补全
