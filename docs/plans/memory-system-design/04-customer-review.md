# 客户对抗审视

> 审视角色：运营经理 / 数据分析师 / 营销专员  
> 审视对象：`02-business-design.md` + `03-product-design.md`  
> 日期：2026-04-02  
> 立场：严格的付费用户，不是内部设计团队

---

## 致命问题（必须修正，否则功能无用）

### 致命问题 1：活动结束后，过时的活动背景没有自动清理机制，只有"降权"

**角色**：营销专员

**问题描述**：

设计中的活动自动降权规则是：`period.end` 距今超过 90 天后，在注入 SYSTEM_PROMPT 时自动跳过。但这意味着：

1. 活动结束后的 90 天内，过时的活动背景仍然会被注入到每次 AI 对话中。
2. 三八大促结束于 3 月 15 日，我 6 月 13 日做 Q2 复盘时，AI 仍然会把这个活动作为"当前背景"提及，把与大促无关的数据波动也往大促上靠。
3. 降权之后的活动只是"不注入"，但没有任何界面告诉我"这个活动已经过期、不再影响 AI"。我根本不知道从哪天起 AI 不再用它了。

**期望行为**：

- 活动结束后第二天就降权（而不是等 90 天），进入"历史归档"状态。
- 归档状态的活动：AI 不主动引用，但在用户明确问"上次大促效果"时才调取。
- `sh memory show campaigns` 必须清楚标注每个活动的当前状态（活跃/已归档）以及归档日期。

**为什么是致命问题**：活动背景注入是营销专员使用记忆系统的核心理由。如果这个机制让 AI 在大促结束后的两三个月里持续用过时背景干扰分析，营销专员会立刻放弃用这个功能，转而手动清理，直接损害核心价值主张。

---

### 致命问题 2：AI 回复的 `[memory: ...]` 标注无法做到"可点击定位"，用户发现错误记忆后的纠正路径太长

**角色**：数据分析师

**问题描述**：

设计中的纠正路径：
```
[memory: 分析洞察·2026-02-16·age=45d]
→ 用户看到这条标注
→ 用户需要手动复制 "2026-02-16" 这个日期片段
→ 执行 sh memory list --search="渠道" 或 sh memory list --stale
→ 从列表中找到对应 ID（可能不是日期，而是 "insight/2026-02-16-channel-conversion"）
→ 执行 sh memory delete insight/2026-02-16-channel-conversion
→ 确认删除
```

这个路径有 4-5 步，需要在脑中将 `[memory: ...]` 的描述文字和 `sh memory list` 里显示的 ID 对应起来。在两者格式不完全一致的情况下（一个是"渠道转化率分析"，另一个是"channel-conversion"），用户极可能找错条目。

**期望行为**：

`[memory: ...]` 标注应该直接包含可以执行的纠正命令：

```
[memory: 分析洞察·2026-02-16·age=45d]
  如有疑问：sh memory delete insight/2026-02-16-channel-conversion
```

或者至少使标注中的 ID 与 `sh memory list` 显示的 ID 完全一致，使用户能够一步复制执行。

**为什么是致命问题**：记忆系统的可信度完全依赖"用户发现错误后能快速纠正"。如果纠正路径复杂到让人嫌麻烦，用户会直接用 `--no-memory` 关掉记忆功能，整个系统价值归零。

---

### 致命问题 3：会话摘要自动写入机制对用户不透明，用户不知道 AI "记住"了什么

**角色**：运营经理

**问题描述**：

设计中 L2 会话摘要是 AI 自动提炼并静默写入的，无需用户确认。这意味着：

1. 每次对话结束后，AI 在后台写了一段"对今天对话的理解"，我完全不知道它写了什么。
2. 下次对话时，AI 说"根据上周的渠道复盘..."，我无法判断 AI 引用的是否是准确的摘要，还是它自己"理解错了"写出来的摘要。
3. 如果 AI 在摘要中写错了结论（比如把"天猫增速高"误写为"京东增速高"），我没有任何机会在它被引用之前发现并纠正。

设计文档中 `sh memory show --used-in-last` 被推迟到 S2（v1.1），MVP 阶段根本没有"查看上次用了哪些记忆"的能力。

**期望行为**：

- 会话摘要写入后，在会话结束时显示一条**低调的**提示："已记录本次会话摘要，sh memory show summary/[ID] 查看内容"——不是强制确认，但让用户知道有东西被记住了。
- MVP 必须包含 `sh memory show summary/[ID]` 的能力（这在设计中已有，不是新增，只需要确保这个是 M 级别而不是 S 级别）。
- 会话 ID 或简短标识应在会话结束时打印出来，让用户能立即查看当次摘要。

**为什么是致命问题**：用户信任是记忆系统的生命线。如果用户不知道 AI 在后台悄悄写了什么、也无法在 MVP 阶段查看，一旦 AI 因为错误摘要给出错误建议，用户会将问题归因为"AI 不可信"而非"有条错误的摘要"，并且没有能力自己排查。

## 重要偏差（影响体验，应该修正）

### 重要偏差 1：冷启动提示文案"约 2 分钟"可能制造心理阻力

**角色**：运营经理

**问题描述**：

冷启动弱提示写道"运行 sh memory init（约 2 分钟）即可开启个性化 AI 体验"。但设计中 `sh memory init` 实际只有 5 个问题，每题平均 10 秒，快速用户 30 秒内就能完成。"约 2 分钟"这个估算偏高，而且在运营经理等忙碌用户眼中，"2 分钟"是一个需要心理建设才能点击的数字。

**修正建议**：改为"约 1 分钟，全部问题可跳过"，更真实，且"可跳过"降低了心理门槛。

---

### 重要偏差 2：`sh memory list` 默认显示 20 条，但用户需要先知道"有什么类型"才能正确使用 `--type` 过滤

**角色**：数据分析师

**问题描述**：

`sh memory list` 不加参数时显示"所有类型混排，按时间倒序，最多 20 条"。这意味着分析师第一眼看到的是一份 profile、context、insight、summary 混杂的列表，并不直觉。

更自然的默认展示应该是"按类型分组"：先显示偏好层摘要（1行），再显示业务上下文摘要（1行），再显示最近3条洞察，再显示最近3条摘要——相当于 `sh memory status` 和 `sh memory list` 的合并视图。

**修正建议**：`sh memory list`（无参数）默认按类型分组展示，而不是按时间混排。`--sort=time` 才是时间倒序模式。

---

### 重要偏差 3：`sh memory add-campaign` 没有"修改已有活动"的入口

**角色**：营销专员

**问题描述**：

活动录入后，活动的效果数据（`effect_summary`）是空的，需要大促结束后补填。但设计中只有 `sh memory add-campaign`（新增）和 `sh memory delete`（删除），没有"编辑已有活动"的命令。

大促结束后我要怎么补填"GMV 环比+15%，转化率+8%"这个实际效果？

现有选项只有：
1. 手动编辑 `~/.socialhub/memory/business_context.yaml` 文件（技术门槛高，不是目标用户的正常操作）
2. 删除旧活动，重新 add-campaign（不合理，还会丢失 created_at 等元数据）

**修正建议**：添加 `sh memory update-campaign --id=ACT001` 命令，或在 `sh memory add-campaign` 中增加 `--edit=ACT001` 参数，支持交互式编辑已有活动的效果数据。

---

### 重要偏差 4：AI 偏好建议机制的用户感知设计存在逻辑漏洞

**角色**：运营经理

**问题描述**：

设计中偏好建议提示是"直接运行下一条命令（而不是输入 y/n/s），自动视为 n（忽略）"。但运营经理在日常使用中很可能这么操作：

1. AI 分析完，显示偏好建议提示
2. 我直接输入下一个分析命令（因为我正在工作流中，注意力在业务上）
3. 提示被自动视为 n

这个行为完全正常，问题在于：**我下次做分析时这个提示还会出现吗？** 设计中 N 之后"30 天内不再为该偏好提示"，但"自动 n"和"主动选 n"的处理规则应该不同——自动 n 应该下次还出现（因为用户可能只是没看到），主动 n 才应该 30 天静默。

**修正建议**：区分"主动选 n"（30 天静默）和"未响应自动忽略"（下次仍可出现，但 7 天内最多出现 1 次）。

---

### 重要偏差 5：PII 扫描命中时"整体丢弃"，用户无感知，但这可能导致有价值的洞察静默丢失

**角色**：数据分析师

**问题描述**：

设计中 PII 扫描命中时"整体丢弃该条记忆写入，用户无感知，仅审计日志可追踪"。这个决策在安全上是对的，但有一个实用性问题：

如果 AI 产出了一条非常有价值的洞察（"Q1 女装品类复购率达 28%，远高于行业均值"），但 AI 在提炼过程中附带提到了一个客户 ID 或手机号（即使是脱敏后的），整条洞察就被丢弃了。

用户完全不知道这条洞察被丢弃，只知道"我明明分析了，但 memory list 里没有"。这会让用户认为记忆系统出 bug 了。

**修正建议**：PII 扫描命中时，在终端输出一条**低调但可见**的提示：`[dim]本次洞察因包含敏感信息，已跳过记录（符合隐私保护规则）[/dim]`，不要完全静默。

## 缺失功能（用户期待但未覆盖）

### 缺失功能 1：活动效果复盘命令的实际体验缺乏定义

**角色**：营销专员

**现状**：设计中多次提到"大促结束后运行 `sh analytics report campaign --id=ACT003` 生成复盘报告"，但 `03-product-design.md` 中完全没有这个命令的设计。

**用户期待**：营销专员最核心的诉求是"活动结束后能快速生成一份包含目标达成情况、关键指标对比、异常原因解释的复盘报告"。这个命令如果只是一个占位符，营销专员使用记忆系统的最大价值点就落空了。

**建议**：要么在产品设计中明确这个命令的输出规格，要么从所有文档中删掉对它的引用，避免用户期待落空。

---

### 缺失功能 2：偏好层缺乏"角色模板快速初始化"

**角色**：运营经理 / 营销专员

**现状**：`sh memory init` 的第 1 个问题是选择角色（运营/分析/营销/管理层），但选择角色后并没有预填相应的默认偏好。用户仍然需要逐一回答后续问题。

**用户期待**：我选择"运营经理"之后，系统应该说"根据运营经理的典型工作模式，已为您预填：默认 7 天、渠道+品类维度、表格输出。您可以修改或直接确认。"——这样"直接回车确认"就能完成初始化，降低 80% 的操作量。

**建议**：为每个角色提供一套预设偏好模板，作为 `sh memory init` 的默认填充值。

---

### 缺失功能 3：跨会话的"未解决问题"追踪缺乏主动提醒

**角色**：数据分析师

**现状**：会话摘要的 `open_questions` 字段存储了"用户尚未决策的问题"（如"Q2 是否需要提升京东渠道投入？"），但下次对话时 AI 不会主动提及这些未解决问题。

**用户期待**：分析师下次启动对话时，AI 应该主动说："上次（3月26日）您还有一个未解决的问题：Q2 是否需要提升京东渠道投入？是否要今天继续分析？"——把 `open_questions` 真正用起来。

**建议**：在会话开始时，如果存在 7 天内的 open_questions，AI 在第一个回复中附带提示。

---

### 缺失功能 4：没有"记忆系统使用说明"的内嵌帮助

**角色**：运营经理（刚接触记忆系统的新用户）

**现状**：文档中描述了大量 `sh memory` 子命令，但对于第一次看到 `sh memory init` 提示的用户，没有一个入口能让他快速了解"记忆系统能帮我做什么、不能做什么"。

**用户期待**：`sh memory --help` 应该在命令列表之上，有 3-4 行人话说明，例如：
```
记忆系统让 AI 记住您的分析习惯和企业业务背景。
 · 第一次使用：sh memory init
 · 查看当前状态：sh memory status
 · 登记营销活动：sh memory add-campaign
```

## 值得保留的设计（明确肯定）

### 肯定 1：冷启动弱提示的非阻断设计非常克制

在没有 user_profile 时，仅在 analytics 命令结果底部显示一次提示（每个进程生命周期最多 1 次），且用户可以通过 `sh memory set __hint_dismissed true` 永久关闭——这个设计完全没有强制引导的压迫感，不会让用户觉得"被逼着做额外工作"。

### 肯定 2：用户显式参数优先级高于记忆偏好，且不触发记忆修改

`sh analytics overview --period=90d` 时，90d 覆盖 `default_period: 7d`，记忆不被修改。这个逻辑天然正确，不需要用户做任何多余操作来"保护"自己的偏好。

### 肯定 3：`sh memory show --injection-preview` 让用户看到"下次对话 AI 会用什么"

这个命令让记忆系统从黑盒变成白盒。用户在重要分析前能提前检查"AI 会带着哪些背景进入对话"，高度满足数据分析师对透明度的需求。

### 肯定 4：洞察的 age 标注设计

`[memory: 分析洞察·2026-02-16·age=45d ⚠️ 超过30天]` 这个格式把时效信息直接嵌入记忆引用标注，用户不需要主动去查洞察的新鲜度——AI 帮他看了。这是一个防止"用户被过时信息欺骗"的好设计。

### 肯定 5：`--no-memory` 临时覆盖机制

不需要修改偏好再改回来。对于运营经理"今天要做季度报告，平时看 7 天"这类高频场景，`--no-memory` 是一个几乎零负担的解决方案。

### 肯定 6：AI 自动写入分析洞察，不需要用户手动操作

L3 洞察层由 insights.py 自动提炼写入，用户什么都不用做。这是记忆系统最接近"自动驾驶"的设计——用户做分析，知识自然积累，零操作成本。

### 肯定 7：偏好建议跨 3 个不同会话才触发，避免临时行为被误判

将触发条件设为"跨 3 个以上不同会话"而不是"单会话内 3 次"，有效过滤了调试性用法和临时偏差，建议质量会更高，用户确认率自然提升。

## 各角色满意度评分（1-10，附理由）

### 运营经理：7 / 10

**理由**：

核心痛点（不用每次重复偏好）的解决方案思路正确，冷启动引导设计克制、不烦人。但失分点在于：

- 会话摘要静默写入不透明（致命问题 3），让我不确定 AI 在我身后写了什么，有点不安心。
- 活动背景 90 天后才降权（致命问题 1），大促后两三个月 AI 还在引用这个活动，会干扰日常分析。
- 缺少角色模板快速初始化（缺失功能 2），`sh memory init` 还是需要逐题回答，不够"一键式"。

如果修正这 3 点，评分可以到 9 分。

---

### 数据分析师：6 / 10

**理由**：

记忆透明度方向是对的（`sh memory list --verbose`、`sh memory show --injection-preview`），但执行上有明显缺口：

- `sh memory show --used-in-last` 被推迟到 v1.1（Should Have），MVP 阶段我无法知道"上次对话究竟用了哪几条记忆"，这是最基础的透明度需求，不应该推迟。
- 发现错误记忆后的纠正路径（致命问题 2）有 4-5 步，且 `[memory: ...]` 标注和 `sh memory list` 的 ID 格式不保证一致，有找错的风险。
- 偏好建议"自动 n"和"主动 n"处理逻辑相同（重要偏差 4），分析师可能在工作流中无意间让提示消失。

总体感受：系统设计了正确的功能，但打磨程度对于追求精确的数据分析师来说还不够，容易在细节上产生不信任感。

---

### 营销专员：5 / 10

**理由**：

活动背景录入（`sh memory add-campaign`）的交互流程设计清晰，操作负担在可接受范围内（5 个问题，2 分钟内完成）。但有两个核心问题让我对这个功能的实用价值存疑：

- 活动效果数据没有编辑入口（重要偏差 3）。大促结束后无法通过命令补填实际效果，只能手动改 YAML 文件，这不是我的工作方式。
- 活动复盘命令（`sh analytics report campaign --id=ACT003`）在设计文档中有多处引用，但完全没有设计规格。这是营销专员最想要的功能，现在是一个空头支票。
- 活动降权设计（致命问题 1）让活动背景在结束后仍干扰分析长达 3 个月，适得其反。

感受：这个功能让我"登记"活动，但不让我"使用"活动效果数据，只做了上半场，下半场缺失。

---

*综合评分：6.0 / 10。方向正确，基础设计扎实，但在用户实际使用的关键路径上有多处断层，需要在进入开发前修正，否则上线后用户投诉的概率很高。*
