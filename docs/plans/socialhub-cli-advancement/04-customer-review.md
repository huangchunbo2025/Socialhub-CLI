# 客户对抗审视

**写作日期**: 2026-03-31
**审视立场**: 三类客户的代言人，挑战一切不以客户价值为中心的设计
**上游文档**: 00-goal.md / 02-business-design.md / 03-product-design.md

---

## 客户 1：电商运营分析师

> **画像**：Excel 熟练，Python 不会，会用 CLI 基本命令。每天早上查销售数据、写报告、分析留存、策划活动。痛点是 BI 系统太慢。关心：查询准不准、速度快不快、结果能不能复制粘贴。

---

### 致命问题

**1. Session 8小时 TTL 打断了分析师实际工作流**

方案把 TTL 设定为 8 小时，理由是"覆盖一个工作日内的分析会话"。这个假设在电商大促场景中直接失效。

真实场景：分析师上午 9 点开始分析 618 备货数据，开完两小时会议，被临时拉去处理客诉，下午 3 点回来想继续追问。距早上 9 点已经 6 小时，Session 还有 2 小时 TTL。但如果她 9 点的 Session 是昨天下午 4 点创建的，今天早上 8 小时到期，她打开终端，Session 已经消失了，一切归零。

更典型的场景：运营报告通常跨天迭代。周一分析流失客户，拿到初步结论，周二开会讨论后需要继续深挖。8 小时 TTL 意味着周一的分析上下文在下班后就消失，周二必须重头开始。

**这不是"设计得不够好"，这是核心用户场景的直接缺失。** 方案中的 TTL 理由基于"分析师会话"的理想模型，没有考虑分析工作被中断的现实。

**2. 结果无法直接复制粘贴到 PPT / Excel**

分析师最关心"结果能不能复制粘贴"。方案里 `--output-format` 只有 `text`、`json`、`stream-json` 三种。

`text` 格式是 Rich 渲染的表格，带有 Unicode 框线字符和 ANSI 颜色代码。复制粘贴进 Excel 或钉钉/飞书时，框线变成乱码，颜色代码变成 `\e[32m` 这样的垃圾字符。

`json` 格式分析师不会用，也看不懂。

**`--output-format csv` 被方案标记为"后续迭代"。** 但对分析师来说，CSV 才是唯一可以直接拖进 Excel 的格式。这是主要用户群最直接的痛点，把它放到后续迭代等于把核心用户价值推迟交付。

**02-business-design.md 第八节已经意识到这个问题**（分析师不会用 JSON），但方案仍然没有把 CSV 纳入 MVP。这是一个已知问题被有意忽略的情况，需要直接追问：为什么？

---

### 重要偏差

**3. Session 提示语仍然技术化，分析师会忽略**

03-product-design.md Round 1 修正了这个问题，Session 提示从 `#a3f2` 改成了：
```
[使用 socialhub -c "..." 继续此对话]
```

但仍然有问题：分析师每次查询后都会看到这条提示，即使她根本不打算继续追问。这条提示出现在每次输出的末尾，长期来看会被当作"噪音"而无视。被无视的功能等于不存在——她不会在需要的时候想到用 `-c`。

**正确的设计**应该是：只在 AI 检测到查询有追问空间时才主动提示（比如"上周数据已分析，如需对比上上周，可继续追问"）。默认不打扰。

**4. 步骤上限提示对分析师造成认知负担**

方案中步骤上限触发时的提示（"AI 生成了 15 个执行步骤，超过安全上限（10步）"）会让分析师困惑：

- 她不知道什么是"执行步骤"
- 她不知道为什么有上限
- "将查询拆分为多条命令"这个建议对不会拆命令的分析师没有实际帮助

这条提示的语言是给技术用户写的。真正对分析师有用的提示应该是："您的查询比较复杂，已为您完成了核心分析。如需更多维度的数据，请分别查询。"

**5. Circuit Breaker 后续建议无法操作**

当 AI 连续 3 步失败触发熔断时，方案提供的建议之一是 `socialhub analytics --help`。但熔断的根因是 AI 生成了不合法的命令序列，分析师看 `--help` 解决不了这个问题。

建议替换为：**自动把失败原因以邮件或钉钉通知发给管理员/技术支持**，而不是让分析师自己排查。分析师的心理模型不包含"调试 AI 命令"这个环节。

---

### 缺失

**6. 没有"把结果分享给同事"的能力**

分析师查完数据，需要把结果发给同事或粘贴到周报。当前方案的所有输出都是终端文本。

缺失：
- 一键把当前查询结果生成为 Markdown/表格 → 复制到剪贴板
- `--export` 输出到文件（`analytics overview --export report.csv`）
- Session 导出为 Markdown（方案标记为"后续迭代"，但这是分析师日常使用中最自然的协作动作）

**7. 没有查询失败时的"为什么没有数据"解释**

分析师查询某个时间段的 VIP 客户，返回空结果。当前方案的空结果处理没有设计（在改进项中未提及）。

分析师看到空结果会困惑：是真的没有数据？还是查询参数写错了？还是权限不够？

缺失一个"空结果诊断"场景的设计：比如"未找到该时间段的 VIP 客户数据，可能原因：①时间段设置错误 ②该分群当期无数据 ③您的账号无该租户数据权限"。

**8. 没有查询历史（可以重新执行上次的查询）**

分析师每天早上重复查看同一批指标。方案里有 `history.json`（被提及但未详细设计），但没有"快速重复上次查询"的命令。

类比：Excel 里的"最近使用的文件"。分析师期望 CLI 能记住"我昨天查的是什么"，方便今天一键重跑。

---

### 肯定

**9. Session 多轮对话方向完全正确**

业务分析（02-business-design.md）关于 Session 的价值判断是准确的：没有多轮对话，CLI 就是"不记得你说过什么的搜索引擎"，产品故事立刻崩塌。这个功能方向正确，是分析师日常触达的核心。

**10. Circuit Breaker 和步骤上限保护了分析师**

分析师不会注意到这个功能，但她会间接受益。防止 AI 幻觉导致大量 API 调用耗尽配额，是对分析师的隐性保护。设计到位。

**11. Session 过期后仍可查看历史记录**

`session show` 可以查看已过期的会话历史，这个设计对分析师有实际价值——她可以回顾"上次是怎么分析的"，即使 Session 无法续接。

**12. 每次 AI 调用后的 Session 提示文案经过了迭代**

Round 1 已经把技术性的 Session ID 从普通用户提示中移除，改为自然语言提示。这个修正方向正确，说明团队有用户感知意识。

## 客户 2：企业 IT 管理员

> **画像**：采购决策者。关心数据安全、合规审计、可控性、出问题能查日志。评估维度：SOC 2、数据不出境、权限控制、依赖库安全。

---

### 致命问题

**1. `config verify-network` 是调试工具，不是采购答案**

方案提供了 `socialhub config verify-network` 诊断命令，用于验证代理和 CA 证书配置是否正确。

IT 管理员在评估工具时问的不是"这个命令能验证网络配置吗"，而是：**"这个工具在我们的企业代理环境下能不能直接用？"**

他们的期望不是"运行一个诊断命令"，而是"工具启动时自动检测代理环境变量，静默通过，我什么都不需要做"。

方案里的 `HTTPS_PROXY` 环境变量自动读取（方式 1）已经覆盖了大部分场景，但在 `verify-network` 命令的设计上花了太多篇幅，反而没有突出"零配置自动适应"才是 IT 管理员真正需要的默认行为。

**IT 管理员不会主动去运行 `verify-network`。** 他们的期望是：工具安装好，代理正常用，如果不通，报错信息里直接告诉他们怎么配置代理（而不是"去运行诊断命令"）。

**2. 日志和审计的位置/格式/保留策略未定义**

IT 管理员在采购审批时，会问：

- "所有操作日志保存在哪里？"
- "日志格式是什么？能导入我们的 SIEM 系统吗？"
- "日志保留多少天？"
- "谁有权限看日志？"

方案中涉及的日志有两类：
- `SecurityAuditLogger`（沙箱违规记录，现有）
- `ai_trace.jsonl`（AI 决策追踪，新增）

但方案没有回答：这两类日志的位置（都在 `~/.socialhub/` 本地，还是有集中化选项？）、格式（能不能被 Splunk/Datadog 消费？）、保留策略（多久轮转一次？保留几个版本？）、访问控制（多用户共享一台机器时，其他用户能不能读到另一个用户的 trace log？）。

这是一个**完整方案的缺失**，不是细节遗漏。IT 管理员看到这里会提出"这个日志方案不完整，我无法评估合规性"。

**3. 没有企业级部署文档和集中配置方案**

方案提到"IT 管理员可以通过以下方式为所有用户预置配置"，给出了两种方式：预置 `config.json` 和设置环境变量。但这只是"能做到"，不是"有文档化的部署方案"。

IT 管理员部署企业软件需要：
- 步骤化的部署 SOP（Standard Operating Procedure）
- 通过 GPO / MDM / Ansible 批量推送配置的参考配置
- 版本升级路径（如何在不破坏用户已有 `config.json` 的情况下推送新版本默认配置）

方案里没有这些。IT 管理员不会因为"技术上能做"就批准采购，他们需要"有标准化文档支撑"的证据。

---

### 重要偏差

**4. `ai_trace.jsonl` 的用户输入明文记录是合规风险**

方案中 `ai_trace.jsonl` 的 `user_input` 字段记录用户的原始输入（明确标注"不脱敏"）。

问题：运营分析师的自然语言输入可能包含客户姓名、手机号、订单号等 PII（个人可识别信息）。比如："查询用户 13800138000 最近三个月的订单"。

这个字段被明文写入本地文件，违反了大多数企业的 PII 处理规范（GDPR / 国内《个人信息保护法》）。IT 管理员在审查这个设计时会直接标记为合规问题，甚至因此阻塞采购。

**修正建议**：`user_input` 默认记录脱敏后的版本（对手机号、邮箱、身份证号等模式进行遮码），或者提供 `ai.trace_user_input=false` 配置项，允许 IT 管理员完全关闭用户输入的记录。

**5. Skills 签名验证的企业审计视角缺失**

方案中 Skills 的 Ed25519 签名验证逻辑清晰，但 IT 管理员关心的不只是"签名有没有验证"，还有：

- "我能不能看到我们企业安装了哪些 Skills 的审计记录？"
- "Skills 的网络访问权限我能不能控制？哪些域名可以访问？"
- "如果某个 Skill 出了安全事件，我怎么快速卸载所有用户的该 Skill？"

沙箱的网络隔离设计（monkey-patch socket）在方案中被提及，但 IT 管理员视角的"Skills 网络访问白名单配置"没有设计。这是一个采购 Checklist 项的缺失。

**6. `ssl_verify=false` 的二次确认还不够**

方案设计了二次确认（输入 "yes" 确认）和执行时警告横幅。这是好的设计方向，但企业 IT 管理员更希望的是：**能通过配置完全禁止 `ssl_verify=false` 选项**（比如企业 IT 可以通过配置或环境变量锁定 `ssl_verify=true`，防止用户绕过）。

方案中没有"管理员锁定安全配置"的能力。

---

### 缺失

**7. 没有多用户/团队共享 CLI 的权限边界设计**

方案假设 CLI 是单用户工具（配置在 `~/.socialhub/`）。但企业环境中，往往是一个服务账号或共享工作站上多人使用。

缺失：
- 服务账号模式（CI/CD 场景，无交互式认证）
- 共享工作站上不同用户数据隔离（每个 OS 用户有独立的 `~/.socialhub/`，这个隐含假设有没有文档化？）
- RBAC（哪些命令需要哪些角色？分析师和管理员权限不同）

**8. 没有离线/断网模式或降级策略**

企业内网环境下，AI Provider（OpenAI/Azure）可能因为合规要求无法访问，或临时断网。

方案没有设计离线降级模式：
- AI 不可用时，直接命令（`sh analytics overview`）是否仍然可以工作？
- 用户看到的错误提示是否足够清晰（区分"AI 不可用"和"数据源不可用"）？
- IT 管理员是否能配置"本地私有化 AI Provider"（Azure OpenAI 私有部署）？

最后一项虽然已有 `AI_PROVIDER=azure` 的支持，但没有针对企业私有化部署的配置文档。

---

### 肯定

**9. 环境变量代理自动读取是正确的零配置设计**

方案中企业代理的"方式 1"（自动读取 `HTTPS_PROXY`/`HTTP_PROXY`/`NO_PROXY`）是 IT 管理员最认可的设计：符合行业标准，已在 `curl`/`pip`/`git` 等工具中广泛使用，大多数企业 IT 已经在全局配置了这些变量。**SocialHub CLI 自动支持，意味着企业内网部署理论上零配置即可工作**。这个方向完全正确。

**10. tenant_id 隔离缓存设计满足数据不出租户要求**

MCP Server 的缓存 key 包含 `tenant_id`，防止跨租户数据泄漏，这是 IT 管理员"数据不出租户"采购 Checklist 的直接满足项。已有的实现值得明确向 IT 管理员展示，而不是埋在代码里。

**11. `ssl_verify=false` 的强制二次确认和运行时警告横幅**

虽然还缺少"管理员锁定"能力，但这个设计已经比大多数 CLI 工具做得好。每次运行时显示警告横幅，防止用户"配置一次就忘了"是真实安全价值。

**12. 输入净化 + 步骤上限回答了 IT 管理员的核心问题**

"AI 会不会执行我不想让它执行的操作？"这是 IT 管理员第一个问题。`[PLAN_START]` 注入修复 + `max_plan_steps=10` 的组合，部分回答了这个问题。能向 IT 管理员展示"AI 执行路径有护栏"。

## 客户 3：Skills 开发者

> **画像**：生态参与者，关心开发体验、文档质量、发布流程、用户反馈。当前痛点：Skills 实际不可用（公钥占位符）。

---

### 致命问题

**1. 密钥占位符问题的用户体验破坏远比方案描述的更严重**

方案用户故事中描述：老张发布了 Skill，但任何用户都无法安装，"开发投入事实上归零"。这是对业务影响的正确描述，但对开发者体验的描述还不够准确。

真实体验链：
1. 开发者花时间开发 Skill，发布到 Store
2. 有用户尝试安装，CLI 报 `SignatureVerificationError`
3. **用户以为是开发者的 Skill 有问题，给开发者打差评/反馈**
4. 开发者花时间排查，最终发现是平台 Bug
5. 开发者在任何公开渠道（GitHub Issues、论坛）发帖，损害平台信誉
6. 其他潜在开发者看到这个问题，放弃进入生态

**方案把这个问题的影响只归结为"开发者体验"，但实际上它是平台信誉危机的引爆点。** 修复的优先级不只是 P0（已标注），而且修复的通知方式也需要设计：所有已发布 Skill 的开发者需要被通知"问题已修复，请验证你的 Skill 现在可以正常安装"。方案没有覆盖这个通知流程。

**2. 开发者第一次发布 Skill 的体验完全没有设计**

方案覆盖了"安装失败的错误提示改进"，但没有覆盖开发者的核心流程：**如何制作一个可以被正确签名的 Skill 包？**

密钥对修复后，私钥在 Skills Store 后端，用于签名。这意味着：
- 开发者开发完 Skill，上传到 Store
- Store 后端用私钥签名
- 用户从 Store 下载时，CLI 用公钥验证

这个流程看起来合理，但方案中没有回答：
- 开发者如何测试"我的 Skill 在本地能不能工作"？（本地没有私钥，无法自签名）
- 开发者如何知道 Store 后端签名成功了？（有没有签名成功的通知？）
- 如果签名失败（比如 Skill 包格式有问题），开发者能看到什么错误信息？

**没有本地测试路径，开发者必须先上传到 Store 才能验证 Skill 是否工作，这是开发体验的重大障碍。**

---

### 重要偏差

**3. 错误提示改进了用户侧，但没有改进开发者侧**

方案改进了用户安装失败时的错误提示（从 `SignatureVerificationError: signature mismatch` 变为有指导意义的多行提示），这是对的。

但对开发者来说，他们需要的错误信息完全不同：
- "你的 Skill 包格式不符合规范（缺少 manifest.json）"
- "你的 Skill 包哈希与 Store 记录不一致，请重新上传"
- "签名服务当前不可用，请稍后再试"

方案完全没有设计 Store 后端对开发者的错误反馈路径。开发者遇到问题，只能靠猜。

**4. Skills 参数化模板（Jinja2）被推迟，但这是开发者生态的关键使能**

02-business-design.md 分析了 Skills 参数化的战略价值：没有参数化，每个客户都需要定制 Skill，无法规模化，护城河不成立。

03-product-design.md 没有包含这个改进（超出 7 项范围），被归为 P2。

但从开发者视角，参数化能力决定了他们的 Skill 是"一次性定制品"还是"可销售的产品"。没有参数化，开发者无法创建"中型零售商通用版本"的 Skill——他们必须为每个客户分别定制，这不是一个可持续的生态。

**5. 开发者文档和 SDK 完全缺失**

方案提到"CRL 吊销检查的实际数据填充（目前框架存在但列表为空）"和"Skills 开发者 SDK"都是后续迭代。

但对开发者来说，**在没有 SDK 和文档的情况下，开发一个 Skill 需要阅读源代码**。这对外部 ISV（零售数据服务商、营销分析公司）是不可接受的。

没有开发者文档，Skills 生态只能是内部团队自用，永远无法吸引外部开发者。

---

### 缺失

**6. 没有本地开发/调试环境**

开发者需要：
- 本地运行 Skill 的方法（跳过签名验证，用于本地测试）
- 本地 Store 模拟器（不连接真实 Store，用于测试安装流程）
- Skill 包验证工具（在上传前检查包格式是否符合规范）

这些工具完全缺失。开发者唯一的测试路径是"上传到真实 Store → 用真实 CLI 安装 → 看结果"，反馈循环极长。

**7. 没有用户反馈回路**

开发者关心用户反馈。当前方案没有设计：
- Skills 的安装/卸载/使用数量统计（开发者看不到自己的 Skill 被多少人用）
- 用户评价/评分机制（Store 页面有没有这个？方案未提及）
- Bug 报告路径（用户发现 Skill 有问题，怎么反馈给开发者？）

没有这些反馈，开发者无法知道自己的工作有没有价值。

**8. 没有版本管理和发布流程的设计**

方案只提到"Skills Store FastAPI 后端"，但没有描述开发者如何发布新版本：
- 如何发布 v1.1.0 版本？是替换 v1.0.0 还是并存？
- 用户已安装 v1.0.0，如何得知有新版本可以更新？
- 有没有灰度发布能力（先发布给 10% 用户测试）？
- 如果新版本有 Bug，如何快速回滚？

---

### 肯定

**9. 签名验证 + 沙箱架构保护了开发者的信誉**

三层沙箱（filesystem/network/execute）和 Ed25519 签名验证，一旦密钥修复，就能确保恶意 Skill 无法进入生态、合法 Skill 不会因为意外操作损坏用户数据。这保护了认真做 Skill 的开发者不被"劣币驱逐良币"。

**10. Skills Store 的基础设施完整**

FastAPI 后端 + React 前端 + JWT 认证 + 开发者账号体系已经建成。一旦密钥修复，开发者可以立即开始发布和测试，不需要等待基础设施建设。对于生态的启动，这是真正的先发优势。

**11. 错误提示改进降低了开发者处理用户投诉的成本**

改进后的安装失败提示（"可能是传输中被篡改或未通过签名"）给出了方向性建议，用户不会立刻去找 Skill 开发者投诉，而是先重试或检查网络。这间接减少了开发者的无效客服工作。

## 综合评判

### 优先级调整建议

方案目前的 P0/P1 排序在业务逻辑上基本正确，但以下几点需要调整：

**必须升级的项目**

| 当前状态 | 改进项 | 调整理由 |
|---------|-------|---------|
| P1（后续迭代） | `--output-format csv` | 分析师主要用户群的核心痛点，放后续迭代等于放弃主要用户 |
| 未提及 | `ai_trace.jsonl` PII 脱敏 | 合规风险，IT 管理员会因此阻塞采购，必须随 trace log 同步设计 |
| 未提及 | 密钥修复后的开发者通知 | 存量开发者不知道问题已修复，不通知等于修复了一半 |

**应该降级或拆分的项目**

| 当前状态 | 改进项 | 调整理由 |
|---------|-------|---------|
| P1 | `socialhub config verify-network` | 对 IT 管理员的真实价值是"静默代理"，不是"主动诊断"。这个命令是辅助工具，不应该作为代理支持的核心设计 |
| P1 | AI trace log 用户可见的 `trace list/show/stats` 命令 | 分析师不会用这些命令，IT 管理员需要的是日志文件的集中化方案，不是单用户 trace 命令 |

**缺失但应当在本版本考虑的项目**

| 优先级建议 | 新增项 | 价值 |
|-----------|-------|-----|
| P1 | Session TTL 可配置（默认 8h，但支持 `config set session.ttl 24h`） | 分析师"第二天继续"的核心场景；实现成本极低，只需一个配置项 |
| P1 | `--output-format csv` 基础实现（不需要全命令覆盖，先覆盖 analytics 系列） | 分析师 Excel 粘贴的刚需；实现成本低（Python csv 标准库） |
| P1 | `ai_trace.jsonl` PII 脱敏配置 `ai.trace_pii_masking=true`（默认开启） | 合规需求，随 trace log 一起设计，不增加额外工作量 |
| P2 | Skills 本地测试模式（`--skip-verify` 仅限 dev 模式） | 开发者生态启动的基础，否则开发者门槛极高 |

### 五个重点追问的裁决

**追问 1：`--output-format json` 分析师会用吗？**

不会。方案已经意识到这一点（02-business-design.md 第八节，03-product-design.md "角色 4"的设计）。JSON 格式是给 IT 自动化工程师和 AI Agent 用的，对分析师无意义。

**问题在于 CSV 格式被推迟了。** 分析师需要的输出格式是 CSV（直接导 Excel），不是 JSON。把 CSV 推到后续迭代，把 JSON 做为 P0，是对主要用户群的价值排序倒置。建议：将 analytics 系列的 `--output-format csv` 与 `json` 同步实现。

**追问 2：Session 8小时 TTL 够吗？**

不够。"第二天继续"的场景是真实的，8 小时 TTL 不覆盖跨天分析。

**建议**：TTL 默认 8 小时，但增加配置项 `session.ttl_hours`，允许 IT 管理员或用户按需调整到 24 小时或更长。实现成本接近零（改一个配置读取点），但解决了分析师跨天分析的核心问题。

**追问 3：企业代理 `config verify-network` 是 IT 管理员想要的吗？**

不是他们第一想要的。IT 管理员想要的是"工具开箱即用，不需要我运行额外命令"。

`verify-network` 是有价值的诊断工具，但它服务的场景是"配置好了但不通"，而不是"默认就能用"。方案的正确重点应该是：**SSL 报错时，错误信息中直接包含代理配置建议**（"如果您在企业网络中，请检查代理设置：`socialhub config set https_proxy <地址>`"），而不是让用户去运行诊断命令。`verify-network` 作为高级诊断工具存在是合理的，但不应该是代理支持设计的主角。

**追问 4：分析师会看 `~/.socialhub/ai_trace.jsonl` 吗？**

绝对不会。分析师不知道这个文件存在，也不知道 JSONL 是什么格式。

`ai_trace.jsonl` 的实际受众是：
- 运维工程师（排查 AI 执行异常）
- 产品负责人（token 消耗归因，方案里的 Chunbo 角色）
- IT 管理员（合规审计，如果他们能集中化读取日志）

对这三类用户，`trace list/show/stats` 命令有价值（运维和产品），日志文件本身有价值（IT 审计）。

**但 `ai_trace.jsonl` 存在一个方案没有解决的问题：它只存在用户本地机器上。** IT 管理员如何集中化读取 100 个用户的 trace log？没有日志集中化方案，审计就是空话。这不是本版本必须解决的，但需要在方案中明确标注这个限制，让 IT 管理员知道当前版本是"本地日志"，未来路径是"集中化上传"。

**追问 5：Skills 密钥修复后，开发者第一次用是什么体验？报错信息够清晰吗？**

密钥修复解决了安装失败的问题，但开发者的第一次使用体验仍然是障碍重重：

- **没有本地测试路径**：开发完必须上传到 Store 才能测试，反馈循环长达数分钟到数小时
- **没有开发者文档**：如何打包 Skill？`manifest.json` 的格式是什么？不读源代码不知道
- **没有签名成功通知**：上传后 Store 后端签名，开发者无法知道签名是否成功
- **用户侧报错改进了，开发者侧没有改进**：当 Skill 出现问题，开发者收到的反馈（如果有的话）仍然是技术错误码，没有指导性

密钥修复是必要的，但修复后的开发者体验仍然是"能用但很痛苦"。建议在修复密钥的同时，至少补充一份最小可用的开发者指南（如何打包、如何测试本地 Skill 包的格式合规性），否则生态仍然只能靠内部团队维持。
